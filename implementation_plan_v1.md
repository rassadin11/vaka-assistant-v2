# Мастер-план реализации: Telegram-сервис персональных ассистентов

Назначение документа: обзор работ, порядок, зависимости и принятые решения. Детализация каждого этапа вынесена в отдельные файлы `plan/stage-N.md` — по файлу на этап. Каждый пункт этапа — самостоятельная задача для отдельного диалога с нейросетью-исполнителем.

Связанные документы:
- `plan/stage-0.md` … `plan/stage-8.md` — детализация этапов.
- `plan/db-schema.md` — зафиксированная схема БД (13 таблиц, для миграции 1.6).
- `tools_registry_v1.md` — спецификация инструментов (закрывает этапы 4–5 по содержанию).
- `plan/pricing-draft.md` — черновик тарифов и юнит-экономики (обсуждение №11).
- `plan/plan-improvements.md` — остаток предложений по улучшению (отложенное/отклонённое; принятое внесено в план).
- `plan/progress.md` — статус выполнения пунктов; отмечается после приёмки.
- `CLAUDE.md` (корень) — протокол работы для любого нового чата: что читать перед задачей, инварианты, регламент.

## Принятые архитектурные решения (контекст для любого исполнителя)

- Multi-tenant: один пул воркеров на всех пользователей; изоляция данных — PostgreSQL RLS по user_id; без per-user контейнеров.
- RLS: `SET LOCAL app.user_id` на каждую транзакцию (session-level SET запрещён — протекает через PgBouncer transaction mode); три роли БД: `migrator` (владелец таблиц, только Alembic), `app` (рабочая, не владелец — иначе RLS обходится), `service` (BYPASSRLS — шедулер, outbox, OAuth-refresh, админ-команды: процессы поперёк пользователей).
- PK: uuid v7 (генерация на приложении, uuid-utils) для сущностей вне аргументов модели; bigint identity для model-facing сущностей (transactions, scheduled_tasks, documents) и логов — DeepSeek надёжнее воспроизводит короткие числа, чем uuid. Детали — plan/db-schema.md.
- Состояние только в хранилищах (Postgres/Redis/S3); гейтвей и воркеры stateless.
- Гейтвей тонкий: secret_token → дедуп → rate limit → enqueue → 200 за <100 мс; без Postgres в hot path; статусы пользователей проверяет воркер.
- Очередь: Redis Streams, партиционирование по user_id, две очереди (interactive / background).
- Redis: два инстанса — `redis-queue` (noeviction, AOF everysec) и `redis-cache` (allkeys-lru, без персистентности). Один инстанс с db0/db1 невозможен: maxmemory-policy задаётся на инстанс.
- Per-user OAuth-токены и прочие программно обновляемые пользовательские секреты — в Postgres с envelope-шифрованием (ключ в vault); vault только для конфигурационных секретов сервисов.
- Telegram: в dev — polling (без публичного URL), в prod — webhook; путь обработки апдейтов общий.
- LLM: DeepSeek через OpenRouter (openai SDK), слой изолирован интерфейсом LLMProvider; prompt caching через статичный префикс. Для тестов — MockLLMProvider, тесты в сеть не ходят.
- Политика данных LLM: в запросах к OpenRouter — `data_collection: deny` (исключаются провайдеры, обучающиеся на данных или хранящие их для обучения, включая официальный DeepSeek API); задаётся в OpenRouterProvider (3.1) и в настройках аккаунта OpenRouter. Причины: в контекст попадают финансы, письма и события календаря; Google Limited Use запрещает передачу данных Workspace API для обучения моделей; передача LLM-провайдерам раскрывается в privacy policy (0.6).
- Голосовые сообщения: STT-слой изолирован интерфейсом STTProvider (+Mock для тестов); транскрипт идёт в обычный агентный путь как текст пользователя. Провайдер — отложенное обсуждение (локальный faster-whisper vs API).
- Эмбеддинги: локальная модель (кандидат по умолчанию bge-m3, 1024 измерения, sentence-transformers); инференс — отдельным внутренним сервисом, а не в каждом воркере (иначе модель дублируется в RAM per-процесс). Сервер рассчитан на это: 16 GB.
- Секреты: vault (Infisical self-hosted); OAuth вместо приёма паролей.
- Платежи: Tribute (webhook-активация подписки).
- Стек: Python 3.12, FastAPI, aiogram 3.x (webhook), asyncio-воркеры, PostgreSQL 16 + pgvector + PgBouncer, Redis, Alembic, Caddy.

Обозначения:
- [DoD] — definition of done, критерий приёмки этапа.
- [Обсудить] — пункт требует отдельного диалога до или во время реализации.

## Список отложенных обсуждений

Нумерация стабильна — на номера ссылаются файлы этапов; дедлайн указан в тексте пункта.

1. Финальная схема таблиц — РЕШЕНО, зафиксирована в plan/db-schema.md.
2. Структура системного промпта — РЕШЕНО на уровне требований: plan/system-prompt-spec.md; текст пишется на 3.4 по спецификации, приёмка — eval-набор из неё же.
3. Выбор хостера сервера и провайдера S3-хранилища бэкапов — разные провайдеры (перед 1B, к концу этапа 3). Решать в связке с юрисдикцией: применимость 152-ФЗ (локализация ПДн граждан РФ) влияет на страну размещения; заявленный GDPR-минимум вопрос не закрывает.
4. Эмбеддинги — РЕШЕНО: локально, сервер 16 GB. На 4.4 остаётся мини-бенчмарк bge-m3 vs multilingual-e5-large на русском (обе 1024 измерения — схему БД выбор не меняет).
5. OAuth-флоу Google целиком: endpoints, refresh, reconnect; хранение решено — шифрованный Postgres (перед 4.7).
6. STT-провайдер: локальный faster-whisper vs API (OpenAI Whisper ~$0.006/мин, Groq whisper-large-v3 дешевле и быстрее). С сервером 16 GB память не блокер; критерии — латентность на CPU (голос — интерактивный ввод) и качество русского. Рабочий вариант: стартовать с API, перейти на локальную за интерфейсом STTProvider (перед 4.9).
7. Правила роутинга моделей и множители бюджетов (перед 8.4, набросок — к 3.6).
8. Маппинг событий Tribute, годовой тариф, запасной канал оплаты, возвраты (refund/chargeback) (перед 7.1/7.4).
9. Сроки хранения данных и автоудаление (перед 7.5).
10. Google-верификация: точный состав scopes и материалы заявки — consent screen, демо-видео, обоснование каждого scope (на этапе 0, до подачи заявки — самый ранний дедлайн списка; верификация занимает недели и лежит на критическом пути к 4.7).
11. Конкретные числа тарифных лимитов и юнит-экономика — ЧЕРНОВИК ГОТОВ: plan/pricing-draft.md (тарифная сетка, модель стоимости сообщения, защита от роста input-токенов). Финал — перед 7.4 по данным usage с беты. Решено: дневные бюджеты в ₽/день вместо токенов (5.2 обновлён). Открыто внутри черновика: комиссия Tribute (№8), цены Pro-моделей (№7), выбор базовой модели deepseek-chat vs V4 Flash (по eval-набору 3.4).

## Регламент передачи задач нейросети-исполнителю

1. В каждый диалог передаются: блок «Принятые архитектурные решения» из этого файла, файл целевого этапа (`plan/stage-N.md`) целиком, DoD этапа, актуальные интерфейсы (файлы) из репозитория.
2. Для этапа 4 дополнительно — разделы 1–3 из tools_registry_v1.md и целевой раздел инструмента.
3. Приёмка любой задачи: тесты зелёные + ручной e2e-сценарий + отсутствие секретов в диффе (gitleaks).
4. Изменения архитектурных решений фиксируются правкой этого файла до реализации, не после; детали этапов — правкой соответствующего plan/stage-N.md.
5. После этапов 2 и 4 — короткая ревизия мастер-плана (~полдня): сроки, вскрывшиеся решения, актуальность отложенных обсуждений.
