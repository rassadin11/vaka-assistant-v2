# Журнал разработки

Делаем Telegram-бота — личного помощника, которому можно написать или наговорить голосом: посчитай траты, напомни о встрече, глянь календарь и почту в Google, найди что-то в интернете, ответь по вопросу из загруженного PDF-файла. Внутри бота работает языковая модель (ИИ, похожий на ChatGPT), которая понимает обычный текст и сама решает, какую функцию бота вызвать — записать трату, поставить напоминание и так далее. Сервис будет платным по подписке, ориентировочно 249 ₽ в месяц.

## 9 июля 2026

Сегодня в основном закладывали фундамент — того, что видно глазами пользователя, пока нет, но без этого дальше двигаться нельзя.

**Документ требований.** Написали и зафиксировали, что бот должен уметь, как он будет зарабатывать (подписка) и как обращаться с данными пользователей — что можно, что нельзя, какие есть обещания по приватности. Это теперь опорный документ: любое решение по проекту сверяется с ним.

**Скелет проекта на GitHub.** Завели репозиторий — место, где хранится весь код проекта с историей изменений. Настроили так, что при каждом изменении кода запускается автоматическая проверка: код смотрят на предмет ошибок, прогоняют тесты (проверки «работает ли то, что должно работать») и отдельно ищут, не попал ли случайно в код пароль или ключ доступа. Первый прогон этих проверок — весь зелёный, то есть без замечаний.

**Регистрации.** Зарегистрировали бота в Telegram (точнее, двух — один для тестов, второй будущий боевой) и получили доступ к сервису, через который бот обращается к ИИ-модели. Все ключи доступа убрали из кода в отдельное защищённое хранилище секретов, чтобы они не могли случайно утечь вместе с кодом.

**Мини-облако для разработки.** Подняли локальное окружение для разработки: одной командой поднимаются база данных, где будут храниться все данные пользователей, очереди сообщений (механизм, который передаёт задачи между частями бота) и хранилище для паролей и ключей. Это уменьшенная копия того, что потом будет работать на боевом сервере — удобно проверять всё локально, не трогая ничего боевого.

**Изоляция данных пользователей — готово.** Сделали так, чтобы один пользователь физически не мог получить доступ к данным другого, даже если в коде где-то ошибка. Работает это на уровне самой базы данных, а не кода бота: завели три «роли» с разными правами доступа. Одна нужна только для изменения структуры базы (создание таблиц и подобного) — с обычной работой бота она не связана. Вторая — рабочая, через неё бот и обслуживает пользователей: если бот открывает базу от имени конкретного человека, база сама показывает только его строки, чужие данные просто не видны, как будто их нет. Третья роль — служебная, для фоновых процессов вроде рассылки напоминаний, которым по смыслу задачи нужно видеть данные всех пользователей сразу.

Чтобы убедиться, что это действительно работает, написали проверку: притворились одним пользователем — база показала только его записи; попробовали обратиться, не сказав, от чьего имени действуем, — база не показала вообще ничего; проверили служебную роль — она видит всё, как и задумано. Проверка прошла успешно.

Отдельно проверили ещё одну вещь — работу «диспетчера подключений» к базе. Это компонент, который экономно раздаёт ограниченное число соединений с базой множеству одновременных запросов от пользователей, чтобы база не перегружалась. В похожих связках раньше встречалась известная ловушка, из-за которой запросы могли внезапно падать с ошибками. Написали проверку специально на этот случай — она подтвердила, что у нас эта ловушка обезврежена.

Обе проверки добавили в тот самый автоматический конвейер на GitHub: теперь при каждом изменении кода поднимается временная копия окружения, и проверки изоляции данных прогоняются сами, без ручных действий. Прогон — зелёный.

**Структура базы данных.** Спроектировали и создали всё «хранилище» бота — 13 таблиц, куда лягут все данные: сами пользователи, история переписки с ботом, короткие выжимки старых разговоров (чтобы бот помнил суть беседы, не перечитывая всё с начала), «память» бота о конкретном человеке, финансовые записи, бюджеты, отложенные задачи и напоминания, загруженные документы и вырезки из них, ключи доступа к Google, очередь исходящих действий бота и два журнала — один фиксирует, какие функции бот вызывал, второй — сколько потрачено на ИИ.

Вся эта структура не создаётся руками — есть автоматический сценарий, который разворачивает пустую базу с нуля одной командой. Уже проверили тестом, что сценарий отрабатывает от начала до конца без ошибок.

Правила изоляции пользователей (те самые, из прошлого раздела — один человек не видит данные другого) включили сразу на всех 13 таблицах, без исключений. И подстраховались тестом на будущее: если кто-то потом добавит новую таблицу и забудет включить для неё изоляцию, эта проверка сама укажет на упущение — прогон перестанет быть зелёным.

Оба журнала (вызовы функций и траты на ИИ) устроили так, чтобы записи автоматически раскладывались по месяцам, а не сваливались в одну общую кучу. За годы работы таких записей накопится очень много, и без такого разделения база со временем начала бы заметно тормозить.

Ещё в паре таблиц — там, где хранится «память» бота о пользователе и вырезки из документов — заранее предусмотрели место под «смысловые отпечатки» текста: специальное представление фразы или абзаца, по которому можно искать похожее по смыслу, а не по точному совпадению слов. Это то, что позже позволит боту находить нужный кусок документа или воспоминание, даже если пользователь спросит о нём другими словами.

Перед тем как принять эту работу, отдельно попросили вторую ИИ-модель перепроверить её независимо — свериться с плановым документом таблица за таблицей, колонка за колонкой. По итогам ревью пару мест укрепили: теперь у каждой записи в базе обязательно должен быть указан её владелец. Это мелочь на вид, но важная — именно благодаря ей потом можно будет полностью и надёжно удалить все данные конкретного пользователя по его просьбе, не оставив хвостов.

**Ключи и пароли пользователей под замком.** Написали механизм, который шифрует ключи доступа пользователей к их Google-аккаунтам (и любые подобные секреты) прямо перед тем, как положить их в базу. Устроено это в два слоя: каждый секрет шифруется своим собственным одноразовым ключом, а этот ключ, в свою очередь, шифруется одним главным ключом, который хранится отдельно от базы данных. Смысл в том, что даже если кто-то посторонний получит копию базы целиком, прочитать эти секреты он всё равно не сможет — ключа для расшифровки там просто нет. Отдельно предусмотрели, что главный ключ можно будет когда-нибудь заменить на новый, не перешифровывая при этом все секреты в базе разом. Механизм проверили тестами, в том числе на «вредные» сценарии: что будет, если кто-то попробует подделать зашифрованные данные, и что будет, если подсунуть не тот ключ для расшифровки — в обоих случаях система корректно отказывает.

**Хранилище секретов запущено.** Развернули отдельный служебный сервис — что-то вроде сейфа, — где теперь живут все пароли и ключи проекта: токен Telegram-бота, ключ доступа к ИИ-модели и так далее. В самом коде этих ключей больше нет вообще, ни в каком виде. Каждая будущая часть системы — приёмник сообщений от пользователей, обработчики их запросов, планировщик напоминаний — получит свой собственный «пропуск» в этот сейф, причём с правом только читать именно то, что ей нужно, и ничего лишнего. Настройку сейфа целиком автоматизировали скриптом: если понадобится развернуть проект на новой машине разработчика, всё необходимое подтягивается одной командой.

**Резервные копии и проверка восстановления.** Настроили автоматическое резервное копирование базы данных: хранится 7 ежедневных копий и 4 еженедельных, и все они зашифрованы. Но главное — бэкап, который никогда не проверяли на восстановление, доверия не заслуживает, поэтому написали отдельный скрипт, который разворачивает копию базы в отдельном «песочном» контейнере (изолированная копия, ничего не задевающая в боевых данных) и сверяет содержимое. Прогнали проверку целиком: заранее пометили тестовую запись, прогнали цикл «бэкап → восстановление» — запись оказалась на месте. Значит, восстановление реально работает, а не только выглядит настроенным на бумаге.

**Организационный момент.** Написание кода теперь по большей части делегировано ИИ-агенту (модель GPT-5.5, через инструмент Codex) — он получает чёткое задание и пишет код, а я (Claude) ставлю ему задачи, проверяю результат тестами и ручной проверкой и принимаю или отправляю на доработку. Так дешевле и не медленнее, но ответственность за качество всё равно на мне.

Итог дня: фундамент готов полностью — окружение для разработки, база данных со всей структурой и изоляцией пользователей, хранилище секретов, шифрование чувствительных данных, резервные копии с проверенным восстановлением. И всё это не разовая ручная проверка, а часть автоматического конвейера — при каждом изменении кода всё перепроверяется само. Дальше — «входная дверь» бота: приём сообщений из Telegram и очередь задач, то есть первая часть, через которую пользователь вообще сможет с ботом заговорить.

## 9 июля 2026 (вечер)

Продолжили строить «входную дверь» бота — очередь сообщений и первую версию воркера, то есть той части, которая эти сообщения разбирает и обрабатывает. Код снова писал ИИ-агент GPT-5.5 (через Codex), Claude ставил задачу и принимал результат.

**Замок на пользователя.** Если человек написал боту несколько сообщений подряд, их может подхватить в обработку не один, а сразу несколько воркеров — а это грозит тем, что ответы перепутаются местами или бот отреагирует дважды на одно и то же. Чтобы так не было, добавили «замок»: пока сообщения конкретного человека обрабатываются, замок на него держит только один воркер, остальные ждут своей очереди. Замок умеет продлеваться, если обработка идёт дольше обычного, и снять его может только тот воркер, который его поставил — чужой замок снять нельзя.

**Что делать, если воркер упал.** Раньше, если воркер вылетал с ошибкой посреди обработки сообщения, само сообщение могло просто потеряться. Теперь так: если сообщение «зависло» без ответа дольше 3,5 минут, его подхватит другой воркер и попробует обработать заново. Если это не получается три раза подряд, сообщение уходит в отдельную «карантинную» очередь — админ проекта получает об этом уведомление, а пользователю бот честно отвечает, что не смог обработать запрос и разработчики уже разбираются, вместо того чтобы промолчать и оставить человека в неведении.

**Приоритеты.** Воркеры теперь всегда сначала разбирают сообщения, на которые человек ждёт немедленного ответа (обычная переписка с ботом), и только потом — фоновые задачи вроде отложенных напоминаний. То есть если система в моменте перегружена, живой разговор с пользователем не встанет в очередь позади фоновой рутины.

**Первый рабочий каркас воркера.** Собрали саму «рабочую петлю»: воркер берёт сообщение из очереди → ставит на пользователя замок → обрабатывает → подтверждает, что сообщение обработано. Пока вместо настоящего ИИ там стоит временная заглушка-эхо (просто повторяет сообщение обратно) — сам ИИ подключим на следующем этапе. Но уже сейчас воркер умеет показывать пользователю «печатает…», пока «думает»; если бота в этот момент перезапускают (например, выкатили обновление), он аккуратно дообрабатывает то сообщение, которое уже начал, а не бросает его на полпути; и переживает временные обрывы связи с Redis (внутренним хранилищем очереди), не теряя данные.

**Найденный баг.** При приёмке Claude обнаружил в коде Codex ошибку и исправил её сам: механизм подбора «зависших» сообщений (тех, что не были обработаны из-за сбоя воркера) на самом деле подхватывал только самое первое такое сообщение из пачки, а остальные молча пропускал — они рисковали уехать в карантин, ни разу не будучи по-настоящему обработанными. Ошибку поправили и перепроверили.

**Проверки.** 49 автоматических тестов прошли зелёным; отдельно вручную прогнали весь сценарий целиком на живом (не тестовом) Redis, чтобы убедиться, что всё это реально работает, а не только в теории; проверили дифф на утечку секретов — паролей и ключей в код не попало.

## 9 июля 2026 (поздний вечер)

Ещё один шаг в строительстве «входной двери» бота — на этот раз про то, чтобы бота нельзя было завалить спамом и чтобы сам бот не нарвался на блокировку со стороны Telegram. Код снова писал ИИ-агент GPT-5.5 (через Codex), Claude принимал результат и поправил только формулировку текста предупреждения пользователю.

**Защита от спама.** Каждый пользователь теперь может отправить боту не больше 20 сообщений в минуту, при этом разрешили небольшой запас — до 5 сообщений подряд «залпом», чтобы не мешать обычной живой переписке. Кто присылает сообщения чаще этого лимита, получает одно вежливое предупреждение «сделайте паузу», а всё, что летит следом сверх лимита, бот просто молча пропускает, не тратя на это ресурсы. Счётчик устроен так, что даже если от одного и того же человека одновременно прилетит куча запросов, обмануть лимит не получится — проверка и увеличение счётчика происходят как одно неделимое действие.

**Вежливость по отношению к самому Telegram.** У Telegram есть свои ограничения на то, как часто боты могут ему писать: примерно 30 сообщений в секунду суммарно и примерно 1 сообщение в секунду в один и тот же чат. Если бот эти правила нарушает, Telegram может на время его придержать. Сделали общий «отправитель» сообщений, который сам держит эти темпы и не даёт боту их превышать. А если Telegram всё же попросит подождать (отвечает ошибкой «слишком часто»), отправитель ждёт ровно столько, сколько сказано, и повторяет попытку — вместо того чтобы просто потерять сообщение. Заодно научили бота автоматически резать слишком длинные ответы на куски, потому что у Telegram есть предел длины одного сообщения (4096 символов). Это не разовая мелочь: без такого «отправителя» первая же массовая рассылка напоминаний (она будет на этапе 4) сразу упёрлась бы в блокировки со стороны Telegram.

Заодно гейтвей (та часть, что принимает сообщения от пользователей) и воркер (та, что их обрабатывает) научились по-настоящему отправлять ответы в Telegram, когда в настройках указан токен бота; а в тестах, где токена нет, они просто записывают, что должны были отправить, в лог — без реальной отправки.

**Проверки.** 58 автоматических тестов прошли зелёным. Отдельно вручную прогнали сценарий на живом Redis: отправили 7 сообщений подряд «залпом» — ровно 5 из них дошли до очереди на обработку, пользователь получил одно предупреждение, а остальное было молча отброшено, как и задумано. Утечек паролей и ключей в код не попало.

## 9 июля 2026 (ночь)

Приняли ещё один кусок «входной двери» бота — на этот раз про то, как в закрытую бету попадают новые люди. Код снова писал ИИ-агент GPT-5.5 (через Codex), Claude ставил задачу и принимал результат.

**Как теперь выглядит вход в бету.** Новый человек пишет боту — бот отвечает, что заявка отправлена, и одновременно уведомляет админа. Админ прямо в том же боте командами решает, кого пускать: `/pending` показывает список заявок, `/approve` и `/reject` — одобряет или отклоняет конкретного человека. Одобренному бот сразу предлагает выбрать свой город кнопками (Москва, Екатеринбург, Владивосток и другие) или написать город текстом — так бот узнаёт часовой пояс человека, чтобы напоминания потом приходили по местному времени, а не по московскому у всех подряд. После выбора города — приветствие с примерами того, что умеет ассистент; команда `/help` в любой момент повторит эти примеры. Отклонённым бот вежливо отказывает, но не чаще раза в сутки, чтобы не заваливать одного и того же человека повторными отказами; забаненным бот не отвечает вовсе. Тексты сообщений пока черновые — финальные формулировки утвердим ближе к запуску беты.

**Найденный при приёмке баг — из прошлого куска.** Ручной сквозной прогон (не автотесты, а именно «руками, как реальный пользователь») поймал ошибку в каркасе воркера, который приняли ещё вечером: когда воркер забирал сообщения сразу из нескольких «отсеков» очереди, он на самом деле обрабатывал только первое, а остальные молча зависали на 3,5 минуты и рисковали по ошибке уехать в карантинную очередь, хотя с ними всё было в порядке. В тестовом прогоне таким образом «потерялась» команда админа. Claude нашёл и исправил ошибку, добавил тест, чтобы она не повторилась незамеченной. Урок отсюда: автотесты были полностью зелёные, а живой прогон всё равно нашёл дыру — поэтому ручная проверка обязательна для каждого принимаемого куска, автотестов одних недостаточно.

**Проверки.** 69 автоматических тестов прошли зелёным. Отдельно вручную прогнали весь сценарий целиком: заявка на бету → одобрение админом → выбор города «самара» → обычный рабочий ответ бота. Утечек паролей и ключей в код не попало.

## 10 июля 2026

Приняли последний пункт этапа 2 — 2.11, «испытания на прочность» всего конвейера приёма сообщений (коммит 4cd909f, автоматические проверки прошли зелёным). Код снова писал ИИ-агент GPT-5.5 (через Codex), Claude ставил задачу и принимал результат. **На этом этап 2 — «входная дверь» бота, гейтвей и очередь — закрыт целиком.**

**Четыре испытания.** Первое — обстрел сотней сообщений в секунду от 20 разных пользователей при 4 параллельно работающих воркерах: ничего не потерялось, ничего не задвоилось, и каждому пользователю ответы пришли строго в том порядке, в котором он задавал вопросы. Второе — «убийство» воркера прямо посреди обработки сообщения: сообщение подхватывает другой воркер, и пользователь получает ровно один ответ, а не ноль и не два. Третье — «ядовитое» сообщение, на котором обработка стабильно падает: после трёх неудачных попыток оно уезжает в отдельную карантинную очередь и не блокирует обработку остальных сообщений. Четвёртое — перезапуск всех воркеров на ходу (так имитировали выкатку новой версии бота в бою): ни одно сообщение при этом не потерялось.

**Самое ценное — найденная дыра в плане.** Первое же испытание вскрыло проблему: если два сообщения одного и того же человека попадали к разным воркерам одновременно, тот воркер, что не успел получить «замок» на пользователя, раньше просто откладывал своё сообщение на потом — и оно могло обработаться позже более свежего сообщения того же человека, то есть ответы бота менялись местами. Починили: теперь воркер сначала убеждается, что все более ранние сообщения этого человека уже обработаны, и терпеливо ждёт своей очереди на замок, вместо того чтобы сразу откладывать сообщение. При приёмке Claude дополнительно ужесточил проверку — насколько глубоко воркер ищет «более ранние» сообщения — и зафиксировал в плане известное ограничение, которое нужно будет учесть на следующем этапе, когда ответы бота станут не мгновенными, а долгими (ИИ иногда думает до 2 минут).

Урок дня тот же, что и вчера: зелёные автоматические тесты — не гарантия того, что всё работает правильно. Дыры находят только испытания, максимально похожие на реальную жизнь.

**Дальше по плану.** Ревизия плана после завершения этапа 2, возврат к отложенным регистрациям (домен, политика конфиденциальности, заявка в Google — они на критическом пути к запуску), затем этап 3 — подключение настоящего ИИ, «агентного ядра» бота.

## 10 июля 2026 (продолжение)

Начался этап 3 — «агентное ядро», то есть тот момент, когда в бота наконец подключается сама ИИ-модель. Приняли первый пункт 3.1 (коммит 71f1f77, автоматические проверки прошли зелёным). Код снова писал ИИ-агент GPT-5.5 (через Codex), Claude ставил задачу и принимал результат.

**Что сделали.** Построили своего рода «розетку» для подключения ИИ: весь остальной код бота теперь общается с языковой моделью через один строгий и единообразный интерфейс, а какая именно модель стоит за этой розеткой — ему знать не нужно. Сегодня в розетку воткнута DeepSeek через сервис OpenRouter, но если завтра захочется попробовать другую модель, переписывать код вокруг не придётся — поменяется только то, что воткнуто в розетку.

**Приватность зашита в каждый запрос.** В настройки каждого обращения к ИИ жёстко прописан явный запрет сервису OpenRouter использовать данные пользователей для обучения своих моделей. Это принципиально: в разговоры с ботом со временем будут попадать финансы человека, содержимое писем, календарь — такие вещи нельзя пускать на обучение чужих моделей ни при каких обстоятельствах. Дополнительно ограничили список компаний, которые физически могут обрабатывать запрос через OpenRouter, — только доверенные из заранее одобренного перечня.

**Счётчик стоимости.** Теперь каждый запрос к ИИ сразу возвращает точную стоимость в долларах. Это фундамент для будущего учёта расходов и тарифных лимитов — без него невозможно будет понять, укладывается ли бот в экономику подписки. Пробный живой запрос к настоящей модели стоил 0,0000024 доллара — четверть тысячной доли цента.

**«Понарошечный» ИИ для тестов.** Сделали заглушку — версию модели с заранее прописанными ответами, которая ведёт себя предсказуемо. Все дальнейшие автоматические тесты агентной логики будут гонять именно через неё: не ходят в интернет, не тратят ни копейки и не зависят от того, отвечает ли настоящий OpenRouter в этот момент.

**Мелкое приключение.** Пока работали, у утилиты Codex CLI на сервере сменилась модель по умолчанию, и старая версия программы из-за этого перестала работать. Обновили утилиту — продолжили без дальнейших задержек.

**Дальше по плану.** Пункт 3.2 — собственно агентный цикл: ИИ получает сообщение от пользователя, сам решает, какие функции бота вызвать (записать трату, посмотреть календарь и так далее), и уже на основе этого собирает финальный ответ.

## 10 июля 2026 (ночь)

Приняли пункт 3.2 — самое сердце ассистента, «агентный цикл» (коммит 4f09d55, автоматические проверки прошли зелёным). Код снова писал ИИ-агент GPT-5.5 (через Codex), Claude ставил задачу и принимал результат.

**Что теперь умеет бот.** Раньше был просто «провод» к ИИ-модели — сообщение туда, ответ обратно. Теперь у ИИ появилась возможность самому решать, что ему нужно сделать, прежде чем ответить. Например, ИИ получает сообщение, понимает, что ему нужно «посмотреть на часы» (то есть вызвать функцию бота, которая знает текущее время), вызывает её, получает результат и уже на основе этого собирает финальный ответ пользователю — а не придумывает время из головы, как обычная языковая модель. Это и есть тот самый механизм, который дальше позволит боту записывать траты, смотреть календарь, ставить напоминания и так далее — каждая такая возможность будет отдельной функцией, которую ИИ сможет вызвать по необходимости.

Первый живой диалог с этим механизмом уже состоялся: тестовый пользователь «жил» в Екатеринбурге, спросил «который час?» — и ассистент действительно вызвал функцию времени и ответил точным екатеринбургским временем (на 2 часа впереди московского), а не общим временем сервера.

**Предохранители, чтобы ИИ не ушёл в разнос.** Добавили несколько ограничителей на одну задачу: не больше 10 вызовов функций подряд, не дольше 2 минут по времени и денежный лимит в рублях — если для ответа потребовалось бы потратить на ИИ слишком много, задача честно останавливается, вместо того чтобы тихо тратить деньги без предела. А если обработка всё-таки затягивается дольше 15 секунд (ИИ иногда «думает» подолгу), пользователь получает промежуточное сообщение «работаю над задачей…», чтобы не смотреть в молчащий экран и не подумал, что бот завис.

**Закрыт долг с прошлого этапа.** Раньше сообщение могло по ошибке уехать в карантинную очередь просто из-за технических переигровок между воркерами, даже если его ни разу по-настоящему не пробовали обработать. Поправили: теперь в карантин уходит только то, что реально трижды не удалось обработать, а не то, что случайно посчитали неудачным.

**Найденный при приёмке баг.** Claude обнаружил, что Codex прогнал перед сдачей только быстрые тесты, а медленные («долгие» испытания, которые проверяют более сложные сценарии) пропустил — и два из них оказались сломаны из-за изменившейся логики. Разобрался и починил: в боевом коде потребовалась ровно одна правка — честный счётчик попыток в карантинной записи, — остальное было обновление самих тестов под новое поведение.

**Организационный момент.** У Codex на сегодня закончился лимит использования. Владелец сказал, что лимит обновится в 3:00 по Москве, и разрешил не ждать ручной команды — поставили запланированную задачу, которая сама в 3:00 возьмёт в работу следующий пункт плана, 3.3 (обработка ситуаций, когда ИИ вызывает функцию бота некорректно — например, с неправильными аргументами).
