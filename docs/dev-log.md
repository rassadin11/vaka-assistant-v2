# Журнал разработки

Делаем Telegram-бота — личного помощника, которому можно написать или наговорить голосом: посчитай траты, напомни о встрече, глянь календарь и почту в Google, найди что-то в интернете, ответь по вопросу из загруженного PDF-файла. Внутри бота работает языковая модель (ИИ, похожий на ChatGPT), которая понимает обычный текст и сама решает, какую функцию бота вызвать — записать трату, поставить напоминание и так далее. Сервис будет платным по подписке, ориентировочно 249 ₽ в месяц.

## 9 июля 2026

Сегодня в основном закладывали фундамент — того, что видно глазами пользователя, пока нет, но без этого дальше двигаться нельзя.

**Документ требований.** Написали и зафиксировали, что бот должен уметь, как он будет зарабатывать (подписка) и как обращаться с данными пользователей — что можно, что нельзя, какие есть обещания по приватности. Это теперь опорный документ: любое решение по проекту сверяется с ним.

**Скелет проекта на GitHub.** Завели репозиторий — место, где хранится весь код проекта с историей изменений. Настроили так, что при каждом изменении кода запускается автоматическая проверка: код смотрят на предмет ошибок, прогоняют тесты (проверки «работает ли то, что должно работать») и отдельно ищут, не попал ли случайно в код пароль или ключ доступа. Первый прогон этих проверок — весь зелёный, то есть без замечаний.

**Регистрации.** Зарегистрировали бота в Telegram (точнее, двух — один для тестов, второй будущий боевой) и получили доступ к сервису, через который бот обращается к ИИ-модели. Все ключи доступа убрали из кода в отдельное защищённое хранилище секретов, чтобы они не могли случайно утечь вместе с кодом.

**Мини-облако для разработки.** Подняли локальное окружение для разработки: одной командой поднимаются база данных, где будут храниться все данные пользователей, очереди сообщений (механизм, который передаёт задачи между частями бота) и хранилище для паролей и ключей. Это уменьшенная копия того, что потом будет работать на боевом сервере — удобно проверять всё локально, не трогая ничего боевого.

**Изоляция данных пользователей — готово.** Сделали так, чтобы один пользователь физически не мог получить доступ к данным другого, даже если в коде где-то ошибка. Работает это на уровне самой базы данных, а не кода бота: завели три «роли» с разными правами доступа. Одна нужна только для изменения структуры базы (создание таблиц и подобного) — с обычной работой бота она не связана. Вторая — рабочая, через неё бот и обслуживает пользователей: если бот открывает базу от имени конкретного человека, база сама показывает только его строки, чужие данные просто не видны, как будто их нет. Третья роль — служебная, для фоновых процессов вроде рассылки напоминаний, которым по смыслу задачи нужно видеть данные всех пользователей сразу.

Чтобы убедиться, что это действительно работает, написали проверку: притворились одним пользователем — база показала только его записи; попробовали обратиться, не сказав, от чьего имени действуем, — база не показала вообще ничего; проверили служебную роль — она видит всё, как и задумано. Проверка прошла успешно.

Отдельно проверили ещё одну вещь — работу «диспетчера подключений» к базе. Это компонент, который экономно раздаёт ограниченное число соединений с базой множеству одновременных запросов от пользователей, чтобы база не перегружалась. В похожих связках раньше встречалась известная ловушка, из-за которой запросы могли внезапно падать с ошибками. Написали проверку специально на этот случай — она подтвердила, что у нас эта ловушка обезврежена.

Обе проверки добавили в тот самый автоматический конвейер на GitHub: теперь при каждом изменении кода поднимается временная копия окружения, и проверки изоляции данных прогоняются сами, без ручных действий. Прогон — зелёный.

**Структура базы данных.** Спроектировали и создали всё «хранилище» бота — 13 таблиц, куда лягут все данные: сами пользователи, история переписки с ботом, короткие выжимки старых разговоров (чтобы бот помнил суть беседы, не перечитывая всё с начала), «память» бота о конкретном человеке, финансовые записи, бюджеты, отложенные задачи и напоминания, загруженные документы и вырезки из них, ключи доступа к Google, очередь исходящих действий бота и два журнала — один фиксирует, какие функции бот вызывал, второй — сколько потрачено на ИИ.

Вся эта структура не создаётся руками — есть автоматический сценарий, который разворачивает пустую базу с нуля одной командой. Уже проверили тестом, что сценарий отрабатывает от начала до конца без ошибок.

Правила изоляции пользователей (те самые, из прошлого раздела — один человек не видит данные другого) включили сразу на всех 13 таблицах, без исключений. И подстраховались тестом на будущее: если кто-то потом добавит новую таблицу и забудет включить для неё изоляцию, эта проверка сама укажет на упущение — прогон перестанет быть зелёным.

Оба журнала (вызовы функций и траты на ИИ) устроили так, чтобы записи автоматически раскладывались по месяцам, а не сваливались в одну общую кучу. За годы работы таких записей накопится очень много, и без такого разделения база со временем начала бы заметно тормозить.

Ещё в паре таблиц — там, где хранится «память» бота о пользователе и вырезки из документов — заранее предусмотрели место под «смысловые отпечатки» текста: специальное представление фразы или абзаца, по которому можно искать похожее по смыслу, а не по точному совпадению слов. Это то, что позже позволит боту находить нужный кусок документа или воспоминание, даже если пользователь спросит о нём другими словами.

Перед тем как принять эту работу, отдельно попросили вторую ИИ-модель перепроверить её независимо — свериться с плановым документом таблица за таблицей, колонка за колонкой. По итогам ревью пару мест укрепили: теперь у каждой записи в базе обязательно должен быть указан её владелец. Это мелочь на вид, но важная — именно благодаря ей потом можно будет полностью и надёжно удалить все данные конкретного пользователя по его просьбе, не оставив хвостов.

**Организационный момент.** Написание кода теперь по большей части делегировано ИИ-агенту (модель GPT-5.5, через инструмент Codex) — он получает чёткое задание и пишет код, а я (Claude) ставлю ему задачи, проверяю результат тестами и ручной проверкой и принимаю или отправляю на доработку. Так дешевле и не медленнее, но ответственность за качество всё равно на мне.
