# Анализ работы бота — 2026-07-17

Источники: SSH-осмотр прода (31.76.15.130), логи gateway/worker, БД (messages, tool_calls_log, users, transactions), Redis-очереди, код репозитория. Стек на момент осмотра полностью healthy, DLQ пустая, застрявших сообщений в очередях нет.

## Статус замечаний владельца

### 1. Бот не читает PDF и фото

Что на самом деле:

- **Фото гейтвей игнорирует молча.** `gateway/app.py::_message_envelope` собирает envelope только для `text`, `voice`, `document`. Сообщение с фото (и его подписью-caption!) не попадает даже в очередь — пользователь не получает вообще никакого ответа. Лог — только `debug` («unsupported message kind ignored»), в проде не виден.
- **PDF-пайплайн есть** (worker/documents.py: инжест → чанки → эмбеддинги → поиск через search_documents), но:
  - **в прод-образе нет tesseract** — проверено (`which tesseract` → пусто). Любой сканированный PDF (без текстового слоя) отвечает «Распознавание сканов временно недоступно». Текстовые PDF обрабатываются.
  - пайплайн — это RAG («Документ обработан, N страниц. Спрашивайте»), а не «вычленить текст и пересказать». Ожидание пользователя «прочитай что тут» покрывается только последующим вопросом через search_documents.

Доработки:
- принимать `photo` в гейтвее: как минимум честный ответ «фото пока не умею», как максимум — OCR/vision-ветка (фото чека → add_transaction — напрашивающийся сценарий);
- caption у фото/документов класть в envelope и передавать модели;
- добавить tesseract (rus+eng) в прод-образ — иначе OCR-ветка 4.6 в проде мертва;
- (опция) режим «выжимка документа»: после инжеста коротко пересказывать содержание, а не только «спрашивайте».

### 2. Иногда не отвечает и нет «печатает…»

Подтверждено кодом; в логах за последние сутки реальных инцидентов не было (стек чист), но механика такая:

- «Печатает…» включается **только когда воркер взял сообщение** (`worker/app.py::_process_with_typing`). Пока сообщение ждёт в очереди (лок пользователя занят, все воркеры заняты, ожидание порядка стрима до таймаута) — обратной связи нет.
- **При ошибке/таймауте обработки пользователю не отвечают ничего.** Исключение логируется, сообщение не ack'ается и ждёт reclaim; до 3 попыток по 120 с (`WORKER_PROCESS_TIMEOUT_SECONDS=120`, `WORKER_MAX_DELIVERIES=3`). Сообщение об ошибке пользователь получает только при попадании в DLQ — спустя минуты. Между попытками typing не идёт — выглядит как «бот молчит».
- Фото и прочие неподдержанные типы (стикеры, локации, media group, edited_message) игнорируются вовсе — см. п.1.

Доработки:
- при таймауте первой же попытки отправлять пользователю «нужно чуть больше времени…»;
- при уходе в ретрай — уведомление или мгновенный честный отказ вместо тихих повторов;
- на неподдержанные типы сообщений отвечать заглушкой;
- метрика/алерт на возраст pending-сообщений в очередях.

### 3. Подтверждение фин. транзакций

В коде подтверждений для `add_transaction` **нет** (RiskLevel.MUTATING_INTERNAL — пишет в БД сразу). Переспрашивает **сама модель**: в истории 2026-07-17 07:42 на «вчера потратил 201 рубль на фастфуд» бот перечислил параметры и спросил разрешения; транзакция так и не записана (в БД всего 1 транзакция).

Доработка: правка системного промпта (v1 и v2-flash): при явном сообщении о совершённой трате/доходе вызывать add_transaction сразу и отвечать фактом записи («Записал: 201 ₽, фастфуд, вчера»), без переспроса; переспрашивать только при реальной неоднозначности (не ясна сумма/направление). Обязательно: новый eval-сценарий + прогон 30/30 (регламент промпт-итераций).

### 4. Календарь-приложение внутри бота

Фичи нет и в плане этапов её нет. Реалистичный вариант — **Telegram Mini App (WebApp)**: страница-календарь на существующем домене (Caddy уже терминирует TLS), auth через initData Telegram, данные — reminders/scheduled_tasks из Postgres, кнопка меню бота. Это отдельный этап: нужен новый HTTP-эндпоинт с проверкой подписи initData, read-only API по данным пользователя (под RLS) и фронт. Требует записи в мастер-план (новый пункт), оценка — заметный, но изолированный кусок работы.

### 5. Модель deepseek-v3 → V4 Flash — ВЫПОЛНЕНО (2026-07-17)

Причина: env `OPENROUTER_MODEL` в проде не был задан, работал дефолт из кода `deepseek/deepseek-chat` (это V3). Сделано по runbook: в Infisical prod досеяны `OPENROUTER_MODEL=deepseek/deepseek-v4-flash` и `OPENROUTER_ALLOWED_PROVIDERS=novita,parasail`, стек перезапущен. Проверка: воркеры логируют `model=deepseek/deepseek-v4-flash, prompt_version=v2-flash`, все контейнеры healthy. Остался плановый хвост: флип дефолта `DEFAULT_OPENROUTER_MODEL` в коде + пересчёт pricing-draft §2/§5 (по плану — к 7.4).

### 6. Не знает текущую дату

Дата **уже передаётся** в каждом запросе: блок USER CONTEXT содержит `Current time: <ISO> (weekday)`, `Timezone`, считается на лету в таймзоне пользователя (worker/agent_processor.py::_dynamics; у владельца Europe/Moscow — корректно). Но в кейсе с Формулой-1 (16.07 22:21 МСК) модель после web_search ответила уклончиво «похоже, сезон 2026», не опершись на известную дату.

Доработка — промпт: явное указание вида «Today is {date}. When the user asks about "next/ближайший" events, resolve against this date, including when summarising web results». Проверить на v2-flash (переход уже случился) + eval-сценарий «следующее событие по дате».

## Найдено дополнительно

7. **Бот выдумывает, что он «GPT-4»** (диалог 17.07 07:41). В промпте нет identity-блока — добавить пару фраз, кто он и на чём работает (без раскрытия внутренностей), либо «не обсуждать модель».
8. **`add_transaction.today_total` смешивает доходы и расходы**: запрос суммы за день не фильтрует `direction` (tools/finance.py, запрос today_total). Модель получает вводящий в заблуждение «итог за сегодня». Однострочный фикс + тест.
9. **Гонка старта с Infisical**: при рестарте стека gateway/worker ~40 секунд сыплют «Infisical request failed: Connection refused», пока Infisical поднимается. Работает за счёт ретраев, но шумит в логах и удлиняет рестарт. Доработка: `depends_on: condition: service_healthy` на infisical в compose.prod.yml (через Ansible-роль).
10. **Потерянные апдейты не видны в проде**: «unsupported update/message kind ignored» — уровень debug. Поднять до warning + счётчик-метрика, иначе п.1/п.2 невозможно диагностировать по логам.
11. **Кейс «82 000 ₽ → доллары»** (16.07): бот записал перевод в инвестиции как транзакцию `other` без отдельной категории инвестиций и вёл диалог про курс с лишним шагом. Когда дойдёт до финансов всерьёз — подумать о категории `investments` в TxCategory (сейчас enum зафиксирован схемой v1 — менять через план).

12. **Markdown в ответах не рендерится** (замечание владельца 2026-07-17): модель выделяет слова `**жирным**`, но сообщения уходят без `parse_mode`, и символы видны как есть. Фикс: конвертер markdown-лайт → Telegram HTML в `core/telegram_sender.py` (жирный/курсив/код/ссылки, экранирование прочего HTML, фолбэк на plain text при ошибке парсинга Telegram).
