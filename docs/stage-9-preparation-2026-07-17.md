## Главное найденное противоречие

Первоначальный план одновременно требовал:

1. резолвить `tg_user_id` в внутренний `users.id` при auth;
2. выполнять DB-доступ только ролью app под RLS;
3. не выдавать webapp роль service.

Обычный RLS-запрос не может выполнить пункт 1: для `SET LOCAL app.user_id` уже
нужен неизвестный UUID. Решение записано до кода: SECURITY DEFINER-функция с
минимальным возвратом `(user_id, status)`, фиксированным search_path, отзывом прав
PUBLIC и точечным EXECUTE для app. Это требует Alembic migration этапа 9, но не
меняет число таблиц и не выдаёт BYPASSRLS приложению.

## Зафиксированные решения

- отдельный webapp container в существующем application image;
- явный prefix `/app`, same-origin, без cookies и CORS;
- itsdangerous bearer session только в памяти frontend;
- status/timezone/plan перечитываются под RLS на каждый API-запрос;
- Preact + TypeScript, SVG-графики без тяжёлого UI/chart framework;
- generation-based invalidation AI-cache вместо Redis wildcard scan;
- один connection-aware domain layer для tools и API;
- текущий X-Frame-Options сохраняется для всего сайта и снимается только на
  `/app/*`;
- backend auth bypass для локальной разработки запрещён.

## Состояние после подготовки

Реализация отсутствует. Тесты приложения не запускались, потому что изменены
