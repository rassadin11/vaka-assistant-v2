## Объём принятой части

Пакет 9.1A подготовил серверную основу Mini App без пользовательских экранов:

- отдельный stateless FastAPI-сервис `webapp`;
- проверка Telegram Mini App `initData` по HMAC и сроку действия;
- выдача подписанной сессии через `itsdangerous`;
- аутентифицированный `GET /app/api/me` с повторной проверкой статуса пользователя;
- RLS-транзакции под ролью `app` и безопасный резолв Telegram ID в внутренний UUID;
- отдельный rate limit Mini App в Redis;
- healthcheck, метрики, trace ID и единый JSON-формат ошибок.

## Зафиксированные решения

- Клиент не передаёт `user_id`: он извлекается только из проверенного Telegram `initData`.
- Связка `tg_user_id → users.id` выполняется единственной `SECURITY DEFINER`-функцией с фиксированным `search_path`, владельцем `migrator`, отозванным доступом `PUBLIC` и разрешением только роли `app`.
- Все пользовательские чтения после аутентификации идут через `SET LOCAL app.user_id` внутри транзакции; session-level `SET` не используется.
- Сессия содержит только версию и внутренний UUID, живёт 12 часов; статус пользователя перечитывается на каждом защищённом запросе.
- Тело `/app/api/auth` запрещает лишние поля. Секрет сессии не может совпадать с токеном Telegram-бота и скрыт из представления настроек.
- Ключи rate limit разделены: существующий gateway сохраняет `rl:user:*`, Mini App использует `rl:webapp:{uuid}`.

## Ручной локальный e2e

В Docker были временно подняты только локальные PostgreSQL, PgBouncer и `redis-cache`. Миграция применена локально. Для двух активных тестовых пользователей с разными timezone/plan выполнен полный путь:

1. сформирован валидный Telegram `initData`;
2. `POST /app/api/auth` выдал отдельную сессию каждому пользователю;
3. `GET /app/api/me` вернул только данные владельца с подтверждённой RLS-изоляцией A/B;
4. после перевода пользователя в `banned` ранее выданная сессия стала получать HTTP 403;
5. тестовые пользователи и Redis-ключи удалены.

## Дефекты, найденные при приёмке

- Устранён конфликт frozen dataclass-исключения с присоединением traceback.
- Добавлен отсутствовавший полный HTTP/RLS e2e для двух пользователей и отзыва уже выданной сессии.
- Для auth-запроса включён запрет неизвестных полей.
- Неожиданные исключения теперь логируются безопасно и без утечки тела запроса или секретов.
