# Этап 5. Фоновые задачи и лимиты (3–4 дня)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

5.1. schedule_task и родственные инструменты (разделы 6.4–6.6 спецификации); минимальный интервал 1 ч.

**Детализация 5.1 (2026-07-11).** Таблица scheduled_tasks из v1 (kind='agent_task'); шедулер-процесс уже есть (4.3) — расширяется, отдельный не заводится.

- **Инструменты** — в `tools/reminders.py` не лезем: новый модуль `tools/scheduled.py`, регистрация `register_scheduled_task_tools(registry, app_pool)` из register_builtin_tools.
  - **schedule_task** (MUTATING_INTERNAL, daily_limit=10): args prompt: str (1–1000), cron: str (≤100), title: str (1–80). Валидация croniter; **минимальный интервал 1 час**: от «сейчас» в таймзоне пользователя берутся 5 последовательных срабатываний croniter — каждый зазор ≥ 3600с, иначе ошибка не-retryable с русским текстом (защита от token-burn). Лимит 10 активных agent_task: COUNT в той же транзакции до INSERT (как 25 у напоминаний). INSERT scheduled_tasks (kind='agent_task', payload=prompt, title, cron_expr, next_run_at = ближайшее срабатывание в UTC, status='active').
  - **list_scheduled_tasks** (READ_ONLY, без аргументов): status='active' AND kind='agent_task', сортировка next_run_at, ≤10 строк; payload: id, title, prompt (усечён до 200), cron, next_run_at (ISO в таймзоне пользователя).
  - **cancel_scheduled_task** (MUTATING_INTERNAL, args task_id: int): UPDATE status='cancelled' WHERE id AND kind='agent_task' AND status='active'; 0 строк → not_found не-retryable. Принадлежность режет RLS.
- **Контракт envelope (правка контракта — фиксируется здесь)**: EnvelopeKind расширяется значением `"agent_task"`. Гейтвей его никогда не создаёт (только шедулер); фильтр гейтвея не меняется.
- **Исполнение** — расширение `worker/scheduler.py`: тик 60с обрабатывает due-строки kind IN ('reminder','agent_task') (FOR UPDATE OF s SKIP LOCKED, как 4.3). Для agent_task вместо отправки текста — enqueue в очередь **background** envelope: kind="agent_task", user_id/chat_id = tg-идентификаторы из JOIN users, payload {"text": prompt, "scheduled_task_id": id, "title": title}, **update_id детерминированный** от (task_id, next_run_at): `10^12 + sha256(f"{id}:{next_run_at unix}") % 10^12` — при переигровке тика после сбоя дедуп воркера (вторая линия) гасит дубль, с реальными Telegram update_id не пересекается (те ~4×10^8). Enqueue и UPDATE next_run_at (croniter от «сейчас» в таймзоне пользователя) — в одной service-транзакции; ошибка enqueue → откат, следующий тик повторит (at-least-once). cron_expr у agent_task всегда есть (one-off нет в v1); status='done' не используется.
- **Обработка в воркере**: envelope kind="agent_task" проходит обычный путь (онбординг-резолв статуса → DocumentRouter → AgentProcessor). AgentProcessor принимает kind="agent_task" наравне с "text" (текст задачи из payload["text"]); атрибуция usage — background (уже так для не-text). Ответ агента уходит пользователю в чат обычным send_reply с префиксом «⏰ {title}:\n» (пользователь понимает, что это фоновая задача, а не ответ на его сообщение). Неактивный пользователь (banned/rejected) — онбординг-слой не пропустит задачу к агенту (существующее поведение); строка задачи при этом остаётся активной до жёсткой обработки в 5.2/этапе 7.
- **Rate limit 2.7 не применяется** (постановка минует гейтвей); дневной бюджет стоимости — 5.2, здесь не реализуется.
- **Тесты**: unit — валидация cron (битый, интервал <1ч включая хитрые выражения типа "*/30 * * * *", валидный ежедневный), лимит 10 активных, list/cancel-маппинг, детерминизм update_id; шедулер с fake-подключениями — due agent_task ставит envelope в background с корректным payload и продвигает next_run_at, ошибка enqueue не продвигает, reminder-ветка не сломана; AgentProcessor принимает agent_task (текст из payload, background-атрибуция); integration — создание/список/отмена под app-ролью, RLS-изоляция, живой тик на реальном Postgres+Redis ставит задачу в очередь. E2e (DoD §6.4): «каждый день в 9 утра собирай сводку моих расходов за вчера» → строка agent_task с корректным next_run_at; принудительный сдвиг next_run_at в прошлое → тик → задача прошла агентный путь с реальным DeepSeek → ответ со сводкой + записи в tool_calls_log/usage(background).

5.2. Дневные бюджеты стоимости per-user в ₽/день (Redis-счётчики; стоимость каждого запроса уже пишется в usage, 3.8); деградация при превышении: отключение background-задач → укороченный контекст → мягкий отказ. Блокировка не применяется. Денежная деноминация вместо токенов — решение из pricing-draft.md §3: устойчивость к росту input-токенов (новые инструменты) и к выбору моделей в Pro. Лимит сообщений/день — производная: бюджет ÷ эталонная стоимость сообщения (эталон пересчитывается из usage ежемесячно).
**Детализация 5.2 (2026-07-11).** Рабочие числа из pricing-draft.md (§4: бюджет 15₽/день у trial/базового; финал — №11 перед 7.4).

- **Учёт трат** — Redis-счётчик в queue-redis `spend_rub:{user_id}:{YYYYMMDD}`, где дата — **в таймзоне пользователя** (DoD этапа: сброс по таймзонам; «сброс» = смена ключа в полночь пользователя, TTL 3 суток при первом инкременте). Инкремент INCRBYFLOAT в воркере в момент записи usage (save_usage/save_stt_usage — через общий хелпер `core/spend.py::add_spend(queue_redis, user_id, timezone, cost_usd)`); конвертация в ₽ по курсу из env `USD_RUB_RATE` (дефолт 90 — допущение черновика). Ошибка Redis не роняет задачу (лог-warning, трата не учтена — fail-open по доступности).
- **Бюджет** — по users.plan из TaskContext: маппинг кодом в `core/spend.py` (trial → 15, standard → 15; будущий pro → 45), env-переопределение `DAILY_BUDGET_RUB_DEFAULT`. Функция `budget_state(spent, budget) -> BudgetState` (enum: ok | no_background | short_context | soft_refuse) с порогами ступеней: **≥100% — no_background, ≥120% — short_context, ≥150% — soft_refuse** (числа рабочие, фиксируются здесь; блокировка не применяется никогда).
- **Ступень 1 (no_background)**: agent_task-запуски не исполняются — проверка в воркере при обработке kind="agent_task" ДО агентного цикла: состояние ≥ no_background → задача не выполняется, пользователю раз в день (Redis-флаг NX EX до полуночи пользователя) уходит «⏰ {title}: фоновая задача пропущена — дневной лимит исчерпан, продолжится завтра»; строка scheduled_tasks не трогается (next_run_at уже продвинут — пропуск одного запуска). Суммаризация диалога (тоже background) продолжает работать — она копеечная и держит контекст здоровым.
- **Ступень 2 (short_context)**: interactive-задачи работают с урезанным контекстом — `build_context` вызывается с флагом lean=True: блок D (память) пропускается, бюджет хвоста истории режется вдвое (детали — параметром в context_manager, контракт блоков не меняется).
- **Ступень 3 (soft_refuse)**: interactive-задача не идёт в LLM — сразу ответ «На сегодня дневной лимит ассистента исчерпан. Продолжим после полуночи — или напишите /feedback, если лимит мешает». Голосовые: STT тоже не вызывается (проверка в VoiceProcessor до скачивания).
- **Порядок проверки**: состояние читается один раз в начале обработки задачи (AgentProcessor/VoiceProcessor); задача, начатая до порога, доводится до конца (перерасход одной задачей допустим — гасится следующей).
- **Тесты**: unit — пороги ступеней (граничные 100/120/150%), ключ по таймзоне (полночь пользователя ≠ полуночь UTC), INCRBYFLOAT+TTL, fail-open при недоступном Redis, agent_task пропуск с одноразовым уведомлением, lean-контекст (блок D пуст, хвост короче), soft_refuse (LLM не вызывается — MockLLMProvider без вызовов), голосовые при soft_refuse (STT не вызывается). Integration не нужен (нет своих таблиц). E2e (DoD этапа 5): искусственно накрутить spend-ключ → все три ступени наблюдаемы вживую (background пропущен с уведомлением, короткий контекст в логах/размере промпта, мягкий отказ), после смены дня (подмена ключа) поведение нормальное.
