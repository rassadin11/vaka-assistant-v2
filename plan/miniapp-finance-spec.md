# Спецификация: экран «Финансы» (Mini App, этап 9.3)

Контекст исполнителю: самодостаточен вместе с plan/stage-9.md (раздел «Ограничения и принципы») и блоком «Принятые архитектурные решения» мастер-плана. Данные — таблицы `transactions` и `budgets` (plan/db-schema.md), существующая логика — `tools/finance.py` (реестр §5), дневные бюджеты ₽ — `core/spend.py` (5.2), LLM — через `LLMProvider`/`ResilientLLMProvider`.

## Назначение

Визуальный дашборд трат: пользователь выбирает период, видит сводку, разбивку по категориям, динамику по дням, прогресс бюджетов, список транзакций — и короткое резюме от нейросети («на что ушло, что изменилось, на что обратить внимание»). Сейчас всё это доступно только вопросами к боту текстом; дашборд делает финансы обозримыми за один взгляд.

## UI (сверху вниз)

### 1. Селектор периода

- Пресеты-чипы: «Сегодня», «Неделя» (текущая, с понедельника), «Месяц» (текущий календарный, по умолчанию), «Прошлый месяц», «Период…» (два date-picker, максимум 366 дней — лимит как у query_transactions).
- Все границы — в таймзоне пользователя. Смена периода перезагружает весь экран.

### 2. Сводка

- Крупно: итог расходов за период; рядом — доходы (если есть).
- Сравнение с предыдущим периодом той же длины: «↓ 12% к прошлому месяцу» (зелёное = расходы меньше, красное = больше; нет данных за прошлый период — не показывать).
- Валюта — ₽ (v1 — RUB, как в transactions.currency по умолчанию; прочие валюты, если появятся, суммируются отдельной строкой без конвертации).

### 3. AI-резюме

- Карточка на 2–4 предложения. Появляется асинхронно (skeleton, отдельный запрос) — не блокирует цифры и графики.
- Состояния: готово / «Резюме появится после первых трат» (пустой период) / плашка «Дневной лимит ассистента исчерпан — резюме будет завтра» (см. API) / скрыто при ошибке LLM (без техтекста).

### 4. Разбивка по категориям

- Donut-диаграмма расходов + легенда-список: категория (русское название + эмодзи: 🍔 food «Еда», 🚕 transport «Транспорт», 🏠 housing «Жильё», 💊 health «Здоровье», 🎭 entertainment «Развлечения», 🛍 shopping «Покупки», 📱 subscriptions «Подписки», 💼 salary «Зарплата», 📦 other «Прочее»), сумма, доля %. Цвета категорий — фиксированная палитра (одна категория = один цвет всегда).
- Тап по категории — фильтрует список транзакций (п.6) и подсвечивает сегмент.
- Доходы в donut не входят (только expense); категория salary встречается только в доходах — в легенде расходов её не будет естественным образом.

### 5. Динамика и бюджеты

- Столбчатый график расходов: по дням (период ≤31 дня) или по неделям/месяцам (длиннее); ось подписана локальными датами.
- Блок бюджетов (если заданы через set_budget): на категорию — прогресс-бар «потрачено X из Y ₽ (Z%)», жёлтый ≥80%, красный ≥100% — пороги как в get_budget_status. Показывается только для периода «Месяц»/«Прошлый месяц» (бюджеты месячные; на прочих периодах блок скрыт).

### 6. Список транзакций

- Постранично (50 на страницу, «показать ещё»), сортировка по ts DESC. Элемент: дата-время локально, категория (эмодзи), описание, сумма (расход красным «−», доход зелёным «+»).
- Активен фильтр по категории из donut — сброс чипом «× Категория».
- Свайп/долгий тап → «Удалить» с подтверждением («Удалить трату 500 ₽, такси, 15.07?»). Удаление — реальный DELETE своей строки; бот при следующем вопросе о тратах её уже не увидит.
- Редактирование — v2 (см. «Отложено»).

## API (webapp, все — под auth 9.1, RLS, rate limit)

### GET /app/api/finance/summary?from=YYYY-MM-DD&to=YYYY-MM-DD

Одним запросом всё, кроме списка и AI-резюме (SQL-агрегаты, никакой выгрузки всех строк в Python):

```json
{
  "period": {"from": "...", "to": "..."},
  "totals": {"expense": "12345.67", "income": "50000.00"},
  "prev_period": {"expense": "14012.00"},        // null если нет данных
  "by_category": [{"category": "food", "expense": "4200.00", "share": 0.34}],
  "by_bucket":  [{"bucket": "2026-07-01", "expense": "310.00"}],  // день|неделя|месяц по длине периода
  "budgets":    [{"category": "food", "limit": "10000.00", "spent": "4200.00", "ratio": 0.42}] // только если период = календарный месяц
}
```

- Суммы — строки (Decimal, квантование как в tools/finance.py), направление считается по `direction`, никогда смешанно (урок бага today_total, п.8 bot-analysis).
- Диапазон >366 дней → 400. Границы — полные локальные дни пользователя, конвертация в UTC на сервере (переиспользовать хелперы границ из tools/finance.py, вынеся их в `core/finance_service.py`).
- prev_period: диапазон той же длины, примыкающий слева.

### GET /app/api/finance/transactions?from&to&category?&cursor?

- Keyset-пагинация по (ts, id) DESC, limit 50; фильтр category опционален. Ответ: `{items: [{id, ts_local, amount, direction, category, description}], next_cursor}`.

### DELETE /app/api/finance/transactions/{id}

- Удаление своей строки; под RLS чужая невидима → 404. Ответ 204. Метрика `webapp_tx_deleted_total`. Инвалидация кэша AI-резюме пользователя (см. ниже).
- В `tool_calls_log` не пишется (не инструмент LLM); у инструментов удаления и нет — это возможность именно Mini App, фиксируется здесь как решение.

### GET /app/api/finance/ai-summary?from&to

Мини-резюме от нейросети.

- **Вход LLM — только агрегаты, не сырой список**: totals, by_category, by_bucket, сравнение с прошлым периодом, топ-5 крупнейших трат (сумма+категория+описание), статус бюджетов. Компактный JSON ≤ ~1500 токенов.
- Промпт (русский вывод): «Ты — финансовый ассистент. По данным о тратах пользователя за период напиши 2–4 предложения: главные статьи расходов, заметные изменения к прошлому периоду, превышения бюджетов. Только факты из данных, без выдуманных цифр, без советов купить финансовые продукты. Без приветствий». Текст промпта — константа в коде webapp/core (это не системный промпт агента; PROMPT_VERSION и eval 3.4 не затрагиваются).
- Модель — базовая (та же, что у агента, через `OPENROUTER_MODEL`); вызов через ResilientLLMProvider (семафор/retry/fallback 3.5). `max_tokens` ≈ 300.
- **Кэш**: redis-cache, ключ `fin_summary:{user_id}:{from}:{to}:{день_в_ТЗ_пользователя}`, TTL 6 ч; принудительная инвалидация — DEL по маске пользователя при add_transaction (в доменной функции) и DELETE из API. Повторное открытие экрана в тот же день — из кэша, без LLM.
- **Учёт и лимиты**: запись в `usage` (queue=background, trace_id запроса) + инкремент spend 5.2. Перед вызовом — проверка дневного бюджета ₽: при ступени ≥100% (no_background) LLM не вызывается, ответ `{status: "budget_exhausted"}` → плашка в UI. Ошибка/таймаут LLM → `{status: "unavailable"}` (кэшировать на 10 минут, чтобы не молотить ретраями).
- Пустой период (0 транзакций) → `{status: "empty"}` без вызова LLM.

## Общая доменная логика (рефакторинг)

Вынести в `core/finance_service.py` из tools/finance.py: границы периодов в таймзоне пользователя, агрегатные запросы, квантование Decimal. Инструменты LLM переключить на эти функции; их поведение и тесты не меняются (регресс-контроль). PNG-график инструмента (matplotlib) остаётся как есть — Mini App рисует своё на фронте.

## Тесты (без сети, MockLLMProvider)

- Юнит: границы периодов (месяц/неделя/произвольный, таймзона не-UTC), prev_period, направление сумм (income не попадает в expense-агрегаты — регресс today_total), share/ratio, выбор гранулярности bucket.
- API: RLS-изоляция на всех эндпоинтах (включая DELETE чужой транзакции → 404); пагинация (граница страниц без дублей/пропусков); ai-summary — кэш-хит без второго вызова Mock, ветки empty/budget_exhausted/unavailable, запись usage и инкремент spend при успехе.

## Приёмка (живой e2e)

1. На живых данных владельца: суммы месяца совпадают с ответом бота на «сколько я потратил в этом месяце» (один источник — сверка вручную).
2. Произвольный период, фильтр по категории, пагинация — работают; donut и столбики соответствуют цифрам.
3. AI-резюме: первый запрос — строка в usage + инкремент spend (проверить в БД/Redis), повторное открытие — из кэша (в usage новой строки нет).
4. Удаление транзакции из Mini App → бот больше не учитывает её в сумме.
5. Запись в progress.md после полной приёмки по регламенту.
