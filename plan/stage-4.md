# Этап 4. Инструменты (7–14 дней)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

Содержание полностью в [../tools_registry_v1.md](../tools_registry_v1.md). Порядок внутри этапа:

4.1. Скелет реестра и диспетчера (раздел 3 спецификации) + тесты контракта.

**Детализация 4.1 (2026-07-10).** Механика подтверждений 4.8 входит в объём 4.1 (pending-ключи, inline-кнопки, outbox-доставка); живая проверка на реальном инструменте — на 4.7, здесь — тестовый mutating_external-инструмент.

- **Контракты (pydantic, `core/tools.py`)** — по коду-скелету §3 реестра: `RiskLevel`, `ToolResult`, спецификация инструмента и `ToolRegistry`. Имя `ToolDefinition` уже занято LLM-контрактом в `core/llm.py` (name/description/json_schema для API) — модель реестра называется **`ToolSpec`** (name, description EN, args_schema: type[BaseModel], risk, handler, daily_limit); `ToolSpec.to_llm_definition()` конвертирует в `core.llm.ToolDefinition` (JSON-схема из pydantic `model_json_schema()`, упрощённая: без $defs/вложенности — инварианты §1.4 проверяются тестом контракта: ≤5 полей, без вложенных объектов глубже 1 уровня).
- **TaskContext расширяется** (контракт в `core/context.py`): добавляются `update_id: int` (из envelope, для ключей идемпотентности §1.7) и `plan: str` (из строки users при онбординг-резолве; для лимитов). Воркер заполняет оба при сборке контекста.
- **Диспетчеризация**: `ToolRegistry.dispatch(ctx, name, raw_args, call_ordinal)` — валидация аргументов pydantic-моделью; `user_id`/поля контекста из аргументов модели игнорируются (лишние поля отбрасываются, `extra="ignore"` недостаточно — поле user_id в схеме инструмента запрещено тестом контракта). Ошибка валидации → `ToolResult(status="error", retryable=True)` с человекочитаемым текстом; **максимум 2 повтора** на (task, tool_name) — счётчик в per-task адаптере, третья подряд ошибка валидации того же инструмента → `retryable=False`.
- **Интеграция с агентным циклом**: per-task адаптер `RegistryToolDispatcher` реализует протокол `ToolDispatcher` из `core/tools_dispatch.py` (definitions → list[core.llm.ToolDefinition], dispatch → JSON-сериализованный ToolResult строкой для модели). Адаптер держит порядковый номер tool-вызова в задаче (для idem-ключа) и счётчики повторов валидации. Семантика MalformedToolCallError из 3.3 не меняется (неизвестное имя/битый JSON — malformed; валидационная ошибка знакомого инструмента — НЕ malformed, это retryable tool_result).
- **Идемпотентность §1.7**: для risk != read_only ключ `idem:{update_id}:{call_ordinal}` в redis-queue, `SET NX EX 86400`; после исполнения — сериализованный ToolResult в значение (`SET XX`); ключ существует со значением → handler не исполняется, возвращается сохранённый результат; ключ существует без значения (упали между NX и записью результата) → `ToolResult(status="error", retryable=False)` c текстом о возможном повторе.
- **daily_limit**: Redis-счётчик `tool_cnt:{user_id}:{tool_name}:{YYYYMMDD}` (дата UTC; смена на таймзону пользователя — при финализации тарифов 7.4), `INCR` + `EXPIRE 86400*2` при первом инкременте, проверка до исполнения; превышение → `ToolResult(status="error", retryable=False)` с русским текстом для пользователя. Счётчики в redis-queue.
- **tool_calls_log**: каждая dispatch-попытка (включая ошибки валидации и отказ по лимиту) пишется через `user_transaction` (RLS): user_id, trace_id, tool_name, args jsonb (каждое строковое значение усечено до 500 символов — §1.3), result_status (ok|error|pending_confirmation), error, latency_ms, created_at. Запись лога не должна ронять задачу: ошибка записи → лог-warning, результат инструмента возвращается.
- **Механика mutating_external (4.8)**: dispatch сериализует вызов в redis-queue `pending:{user_id}:{uuid}` (TTL 900с; payload: tool_name, validated args, update_id, chat_id, trace_id), шлёт пользователю сообщение с описанием действия и inline-кнопками «Выполнить» / «Отменить» (callback_data `confirm:{uuid}` / `cancel:{uuid}`), возвращает `ToolResult(status="pending_confirmation")` — модель завершает ход текстом «ожидаю подтверждения». Callback в gateway → envelope kind=callback (расширение существующего kind, если ещё нет) → воркер: читает pending-ключ (нет ключа → «действие устарело»), «Отменить» → удалить ключ + уведомить; «Выполнить» → INSERT в outbox_actions (status=pending) через user_transaction + удалить pending-ключ. **Outbox-процессор** — фоновая задача воркера под ролью `service` (кросс-пользовательский SELECT по (status, created_at)): берёт pending-действия, помечает executing, исполняет handler инструмента с восстановленным TaskContext, пишет done/failed (+attempts, last_error, ≤3 попыток), уведомляет пользователя о результате. В 4.1 исполнение проверяется тестовым инструментом.
- **Профили §4 в 4.1 не реализуются** (инструментов ещё нет) — `get_for_context(ctx, connected)` в v1 возвращает все зарегистрированные; профили включаются по мере появления инструментов (4.2+).
- **Тесты (DoD §3)**: инжекция user_id (аргумент модели игнорируется), невалидные аргументы → retryable error + лимит 2 повторов, ветка pending_confirmation (ключ в Redis, кнопки, callback-подтверждение → outbox → исполнение), daily_limit, идемпотентность (повторный dispatch с тем же ключом не исполняет handler), запись в tool_calls_log при всех статусах; интеграционные — на живом Redis/Postgres как в tests/integration/.
4.2. Финансы (раздел 5) — обкатка контракта без внешних зависимостей.

**Детализация 4.2 (2026-07-10).**

- **Миграция не нужна**: transactions и budgets созданы миграцией v1 (db-schema.md, 13 таблиц); пункт DoD §5 «миграция таблиц» считается выполненным этапом 1.6.
- **Модуль `tools/finance.py`**: 4 инструмента по §5 (add_transaction, query_transactions, set_budget, get_budget_status). Регистрация — функция `register_finance_tools(registry, app_pool, send_photo)`: зависимости хендлеров (пул, колбэки) передаются при регистрации замыканиями; `tools/registry.py::register_builtin_tools` расширяется параметрами зависимостей (воркер прокидывает).
- **Все записи/чтения — через `user_transaction`** (RLS, user_id из TaskContext). Суммы: float из аргументов модели конвертируется в Decimal с квантованием до 0.01 до любых проверок; amount ≤ 0 → ошибка retryable.
- **Валюта**: в v1 аргумента currency нет (лимит 5 полей §1.4), все записи в дефолтной 'RUB'; мультивалютная группировка в query_transactions реализуется по колонке currency (задел на будущее), для RUB-only данных вырождается в обычную.
- **Время**: ts из аргументов — ISO 8601; naive-строка интерпретируется в users.timezone, aware — как есть; хранение timestamptz (UTC). ts в будущем > +1 суток от «сейчас» → ошибка (не retryable). period_start/period_end в query_transactions — ISO-даты в таймзоне пользователя, границы [00:00 start; 24:00 end) в этой таймзоне; period_end < period_start или период > 366 дней → ошибка.
- **query_transactions**: агрегация SQL-ом (SUM по группам), group_by ∈ {category, day, none}; максимум 100 строк payload (усечение с флагом truncated). График: при group_by="day" и периоде ≥ 14 дней хендлер рендерит PNG (matplotlib, backend Agg, зависимость в worker-группу) и отправляет через колбэк `send_photo(chat_id, png_bytes, caption)` fire-and-forget (ошибка отправки не ломает tool_result); `core/telegram_sender.py` дополняется методом send_photo (sendPhoto multipart, те же лимиты/ретраи, конвенция как у send_message). В tool_result график не входит (§5.2).
- **Бюджетный триггер**: add_transaction после INSERT считает расход месяца по категории (границы месяца в users.timezone); payload содержит today_total по категории и, при наличии бюджета, budget_status {monthly_limit, spent, percent}; при пересечении порогов 80%/100% (расход до вставки был ниже порога, после — выше либо равен) payload дополняется полем budget_alert с русским текстом для реплики. Уведомление — только в payload (модель озвучивает), отдельного push нет.
- **set_budget**: UPSERT budgets (PK user_id+category), monthly_limit квантуется до 0.01, ≤ 0 → ошибка retryable. **get_budget_status**: без аргументов; все бюджеты пользователя + расход текущего месяца (таймзона пользователя) в процентах.
- **daily_limit** (§12 не задаёт числа для финансов): v1 — без лимитов (None) у всех четырёх; лимиты появятся с тарифами (7.4).
- **Тесты**: unit на конвертацию/границы (amount, ts в будущем, период 366, квантование), агрегации и группировки (включая мультивалютную ветку и усечение 100 строк), бюджетные пороги 80/100 (пересечение и непересечение), график (вызов send_photo при day+14д, отсутствие при коротком периоде; PNG-байты непустые); integration — insert/select под app-ролью, RLS-изоляция, UPSERT budgets. E2e по DoD §5: «потратил 500 на такси» → запись в transactions + запись в tool_calls_log.
4.3. Напоминания (6.1–6.3) + процесс-шедулер (тик 60 с, croniter).

**Детализация 4.3 (2026-07-10).**

- **Объём**: только 6.1–6.3 (kind=reminder) + шедулер-процесс. schedule_task/6.4–6.6 (kind=agent_task) — этап 5; шедулер строки agent_task не трогает (фильтр kind='reminder').
- **Модуль `tools/reminders.py`**, регистрация `register_reminder_tools(registry, app_pool)` из register_builtin_tools. Миграция не нужна — scheduled_tasks из v1.
- **create_reminder** (MUTATING_INTERNAL, daily_limit=30): remind_at ISO 8601, naive → users.timezone; remind_at ≤ сейчас → ошибка retryable с текущим временем пользователя в тексте. repeat→cron_expr маппинг по локальному времени remind_at в таймзоне пользователя: daily `M H * * *`, weekly `M H * * DOW`, monthly `M H DOM * *`, none → NULL. INSERT scheduled_tasks (kind='reminder', payload=text, title=усечённый payload ≤80 символов, next_run_at=remind_at в UTC, status='active'). Лимит 25 активных: COUNT в той же транзакции до INSERT; превышение → ошибка не-retryable с русским текстом.
- **list_reminders** (READ_ONLY, без аргументов): status='active' AND kind='reminder', сортировка next_run_at, ≤25 строк; payload: id, text, next_run_at (ISO в таймзоне пользователя), repeat (обратный маппинг из cron_expr: none|daily|weekly|monthly|cron).
- **cancel_reminder** (MUTATING_INTERNAL): UPDATE status='cancelled' WHERE id=$1 AND kind='reminder' AND status='active'; принадлежность режет RLS; 0 строк → ошибка not_found (не retryable).
- **Шедулер** — `worker/scheduler.py`, фоновая asyncio-задача воркера под ролью service (как OutboxProcessor), тик 60с. Семантика at-least-once: в одной service-транзакции SELECT ... WHERE kind='reminder' AND status='active' AND next_run_at <= now() ORDER BY next_run_at FOR UPDATE SKIP LOCKED LIMIT 20 → для каждой строки JOIN users (tg_chat_id, timezone) → отправка «Напоминание: {payload}» через telegram sender → cron_expr IS NULL: status='done'; иначе next_run_at = croniter(cron_expr, база — now() в users.timezone).get_next() → UTC, last_run_at=now(). Ошибка отправки → откат транзакции (row lock снимается, следующий тик повторит); повторная доставка при гонке исключена SKIP LOCKED. Несколько реплик воркера безопасны за счёт row-lock.
- **croniter** — новая зависимость (uv). Пересчёт всегда от текущего момента в таймзоне пользователя (DST-переходы обрабатывает croniter).
- **Тесты**: unit — маппинг repeat→cron и обратно, remind_at в прошлом/naive-интерпретация, лимит 25, усечение title; шедулер с fake-подключением — one-off → done, recurring → next_run_at строго в будущем, ошибка отправки не помечает done; таймзоны — границы суток (23:30 Europe/Moscow = 20:30 UTC; переход месяца); integration — создание/список/отмена под app-ролью + RLS-изоляция, живой тик шедулера на реальном Postgres (due-строка доставляется, недоставленная при падении sender остаётся active). E2e: «напомни завтра в 9 купить хлеб» → корректная строка в scheduled_tasks (время в таймзоне пользователя) + запись в tool_calls_log; принудительный тик доставляет напоминание.
4.4. Память (раздел 7): эмбеддинги локальные (решено, обсуждение №4). Мини-бенчмарк bge-m3 vs multilingual-e5-large на русском (20–30 пар «запрос → релевантный факт»), фиксация победителя. Инференс — отдельный внутренний сервис: одна копия модели в RAM, воркеры ходят по HTTP/unix-socket (в каждом воркере модель не поднимать — умножается на число процессов). Автоинжект top-5, дедупликация. Vector-колонки и HNSW уже созданы миграцией v1. Недоступность сервиса эмбеддингов не роняет задачу: блок D (факты) пропускается с записью в лог, remember_fact возвращает retryable-ошибку; алерт — в 6.4.
**Детализация 4.4 (2026-07-10).** Порядок: бенчмарк → фиксация победителя правкой этого файла → сервис → инструмент → автоинжект.

- **Бенчмарк** — `evals/embeddings_benchmark.py` + датасет `evals/data/memory_bench_ru.json` (вне pytest/CI, как prompt_eval): 25–30 пар «запрос → релевантный факт» на русском + пул ≥50 фактов-дистракторов (реалистичные личные факты: работа, семья, здоровье, предпочтения, финансы). Обе модели через sentence-transformers (у e5 обязательные префиксы query:/passage:, у bge-m3 — без). Метрики: recall@1, recall@5, MRR; победитель по recall@5, тай-брейк MRR → результат фиксируется здесь же в плане. **Результат (2026-07-10): победитель multilingual-e5-large** — recall@5 0.929 / MRR 0.857 / worst_rank 13 против 0.893 / 0.883 / 51 у bge-m3 (по критерию плана recall@5 решает; хвост у e5 заметно стабильнее). Калибровка порогов на датасете: порог автоинжекта 0.80 (p99 нерелевантных 0.801; 0.55 из реестра не работает для e5 — реестр §7.2 поправлен), дедуп 0.92 валиден (разные факты ≤0.885). Запуск моделей: `HF_HOME` на D: (C: переполнен), transformers<5 (bge-m3 без safetensors). Зависимости бенчмарка и сервиса (sentence-transformers, torch CPU) — отдельная uv-группа `embeddings`, в основные зависимости и CI не входит.
- **Сервис эмбеддингов** — новый топ-уровень `embeddings/` (FastAPI, как gateway): одна копия модели-победителя в RAM; `POST /embed {"texts": [...], "kind": "query"|"passage"}` → `{"vectors": [[...1024...]]}` (kind управляет префиксами e5; для bge-m3 игнорируется), `GET /healthz`. Слушает только внутренний интерфейс (в dev — localhost-порт в docker-compose; модель кэшируется в volume). Батчи ≤64 текстов, текст усечён до 512 токенов модели.
- **Клиент** — `core/embeddings.py`: протокол `EmbeddingsProvider` (embed(texts, kind) -> list[list[float]]), `HttpEmbeddingsProvider` (httpx, таймаут 5с, без ретраев — недоступность обрабатывают вызывающие), `MockEmbeddingsProvider` для тестов (детерминированные векторы). URL из env `EMBEDDINGS_URL`; пустой → провайдер None и функциональность памяти отключена (лог-warning при старте).
- **tools/memory.py** — `remember_fact` (MUTATING_INTERNAL, daily_limit=50): эмбеддинг факта (kind=passage) → в одной user_transaction: ближайший существующий факт по cosine (`embedding <=> $1`, pgvector); similarity > 0.92 → UPDATE updated_at/last_used_at существующего (payload {deduplicated: true}); иначе INSERT (uuid7); затем COUNT > 500 → DELETE самых старых по last_used_at сверх лимита. Сервис недоступен (ConnectError/Timeout/5xx) → ToolResult error retryable=True («память временно недоступна»). Вектор в pgvector передаётся строковым литералом '[...]' (или через pgvector-python) — решить по стилю репо.
- **Автоинжект (блок D)** — в worker/agent_processor.py перед build_context: эмбеддинг текста задачи (kind=query) → top-5 memory_facts по cosine similarity ≥ 0.80 (порог откалиброван под e5, см. результат бенчмарка) (SELECT под user_transaction, `1 - (embedding <=> $1) AS sim`) → тексты фактов в параметр facts (блок D контекст-менеджера, бюджет 400 уже есть); у заинжектированных фактов UPDATE last_used_at (та же транзакция). Сервис недоступен → facts=() + лог-warning, задача НЕ падает (алерт — этап 6.4). provider None → тот же путь.
- **Воркер**: HttpEmbeddingsProvider создаётся при непустом EMBEDDINGS_URL; передаётся в AgentProcessor и register_builtin_tools (memory-инструмент регистрируется только при наличии провайдера).
- **Тесты**: unit с MockEmbeddingsProvider — дедуп-ветка (>0.92), вставка, вытеснение сверх 500, retryable при недоступности, автоинжект (порог 0.55, top-5, обновление last_used_at, пропуск при недоступности); integration — pgvector-запросы на живом Postgres (вставка вектора, cosine-поиск, HNSW-индекс используется хотя бы EXPLAIN-ом не проверяем — функционально). E2e (DoD §7): remember_fact живьём через сервис → факт в БД; повторный похожий факт → дедуп; новый диалог «что ты обо мне помнишь про <тему>?» → факт в блоке D и в ответе.
