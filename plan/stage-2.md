# Этап 2. Гейтвей и очередь (4–6 дней)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

2.1. FastAPI-приложение gateway (тонкое, см. «Принятые решения»): endpoint webhook, проверка secret_token; /healthz (для Caddy и деплой-гейта); /metrics-заглушка. aiogram 3.x: allowed_updates явным списком (message, callback_query; pre_checkout_query — задел); set_webhook — отдельной командой, не на старте приложения (иначе dev/prod воюют за webhook). Dev-режим — polling, путь обработки апдейтов общий с webhook. Группы вне скоупа v1: гейтвей отбрасывает апдейты с chat.type != private до enqueue (иначе добавленный в группу бот жжёт токены на чужих сообщениях); у бота в BotFather запрещается добавление в группы.
2.2. Дедупликация update_id (redis-cache, `SET dedup:{update_id} NX EX 86400`). Инвариант порядка: дедуп-ключ ставится ПОСЛЕ успешного enqueue (иначе падение между SETNX и enqueue теряет апдейт — Telegram ретраит, но дедуп его срежет); вторая линия — дедуп по update_id в воркере. Инвариант оформлен тестом.
2.3. Очередь Redis Streams: 16 партиций на очередь (`q:interactive:{0..15}`, `q:background:{0..15}`), партиция = crc32(user_id) % 16 (менять без ребаланса нельзя — заложить с запасом). Одна consumer group на очередь; воркеры читают все партиции. Схема сообщения (envelope) — pydantic-модель в core/, контракт между этапами 2 и 3: {update_id, user_id, chat_id, kind, payload, trace_id, enqueued_at, attempt}. kind: text | voice | document | callback; для voice/document гейтвей кладёт в payload только tg_file_id и метаданные (duration, size) — скачивает файл воркер, гейтвей остаётся тонким.
2.4. Per-user lock: `SET lock:user:{id} {worker_token} NX PX 180000`; продление при долгих задачах (этап 3: LLM-цикл до 120 с); освобождение только владельцем (check-and-del Lua). Воркер, не взявший лок, возвращает сообщение без ACK.
2.5. Redelivery и DLQ: XAUTOCLAIM по idle > 3.5 мин (заведомо больше лока 180 с и таймаута задачи 120 с — легитимная задача не переигрывается, а после падения воркера пользователь ждёт меньше); счётчик доставок в envelope; после 3 доставок → q:dlq + уведомление админу. Пользователю при уходе его задачи в DLQ — честное сообщение («не получилось обработать запрос, разбираемся»), иначе бот для него молча умирает. Без DLQ ядовитое сообщение навсегда останавливает партицию.
2.6. Приоритеты: воркер опрашивает interactive (block=0), при пустоте — background (block=2s); параметр «доля воркеров только-interactive» — защита латентности при завале фоном.
2.7. Rate limit per-user (token bucket, Lua-скрипт в redis-cache — атомарность): 20 сообщений/мин, burst 5. При превышении: одно вежливое предупреждение, дальше молчаливый дроп с TTL-флагом «уже предупредили». Доставка предупреждения: гейтвей сам сообщений не шлёт — использует общий Telegram-sender из 2.8 fire-and-forget-задачей (asyncio.create_task), hot path не блокируется; если у процесса гейтвея нет токена бота (тесты), предупреждение только логируется.
2.8. Отправка ответов: общий rate-limiter отправки (лимиты Telegram: ~30 msg/s глобально, ~1 msg/s на чат) + retry на 429 по retry_after. Упрётся в это первый же массовый broadcast напоминаний (4.3) — закладывается здесь.
2.9. Онбординг беты: /start → users.status=pending → уведомление админу → approve/reject. Статусы: pending / active / rejected / banned. Админ-интерфейс — тот же бот, команды по списку admin tg_id из конфига (/pending, /approve <id>, /reject <id>); отдельный админ-бот не нужен. Таймзона при активации: инлайн-кнопки популярных городов + fallback «пришли город текстом»; хранится IANA-имя. Проверка статуса — в воркере, не в гейтвее. Активация сопровождается welcome-сообщением (что умеет ассистент, 3–5 живых примеров запросов); /help повторяет его в любой момент. Тексты здесь — черновые; финальные — к закрытой бете (DoD этапа 4).
2.10. Каркас воркера (echo — временный плейсхолдер process()): цикл consume → lock → process → ack; на время process — sendChatAction «typing…» циклом (~каждые 5 с), чтобы пользователь видел, что запрос обрабатывается; graceful shutdown по SIGTERM (доделать текущее, ACK, отпустить лок — иначе каждый деплой роняет задачи); реконнекты к Redis; structured-лог с trace_id из envelope. Всё, кроме echo, переживает этап и несёт этап 3.
2.11. Тесты контура:
  - Нагрузочный (k6/locust, фейковые апдейты в webhook): 100 msg/с; отправлено == обработано; нет дублей (по update_id); порядок per-user не нарушен (монотонный счётчик в тексте).
  - Chaos: kill -9 воркера посреди обработки → сообщение переигрывается, не теряется, не дублируется наружу.
  - Ядовитое сообщение → уезжает в q:dlq, партиция продолжает обрабатываться.
  - Рестарт всех воркеров (имитация деплоя) — ни одна задача не потеряна.

[DoD] Все тесты 2.11 пройдены; последовательность сообщений одного пользователя не нарушается при 4 воркерах.
