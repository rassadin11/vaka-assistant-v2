# Онбординг-скрипт, итерация 1: приветствие + капельные подсказки

Статус: утверждено владельцем 2026-07-18 (тексты согласованы в диалоге; акценты: возможности ассистента + голосовой ввод, предупреждение про длительность веб-поиска). Внеплановая задача, учёт — в progress.md «Внеплановые задачи».

## 1. Цель

Первый контакт пользователя с ботом должен: (а) продать ключевые возможности, особенно голосовой ввод; (б) довести до первого успешного действия за 30 секунд (готовые фразы-примеры); (в) дозировать обучение — подсказки о смежных возможностях приходят после первого реального использования функции, а не стеной текста в начале.

## 2. Финальные тексты

Все тексты используют markdown-лайт (`**жирный**`) и отправляются через существующий конвертер markdown → Telegram HTML (`core/telegram_format.py`, фолбэк plain при ошибке парсинга). Тексты ниже — канонические; при реализации переносить байт-в-байт (включая эмодзи и кавычки-ёлочки).

### 2.1 Первое касание: без ручного одобрения (решение владельца 2026-07-18)

Ручное одобрение заявок отменено — новый пользователь активируется сразу (досрочное выполнение пункта 7.7 мастер-плана в части механизма; пункт 2.9 «онбординг беты» в части гейта утрачивает силу). `APPLICATION_RECEIVED_TEXT` и `PENDING_REPEAT_TEXT` удаляются. Первый `/start` (и вообще первое сообщение неизвестного пользователя) отвечает одним сообщением с кнопками городов — `START_TIMEZONE_PROMPT_TEXT`:

```
Привет! Я — персональный ассистент: расходы, напоминания, поиск в интернете, PDF-документы — всё обычными словами в чате, текстом или голосом.

Чтобы напоминания приходили вовремя, выберите ваш часовой пояс — по разнице с Москвой:
```

Уведомление админам о новой заявке заменяется информационным: `Новый пользователь: id {tg_user_id}.` (без призыва /approve).

### 2.1a Кнопки часовых поясов (обновлено 2026-07-20, решение владельца)

Прежняя раскладка кнопок (8 городов вперемешку: Москва, Петербург, Калининград, Екатеринбург, Новосибирск, Владивосток, Минск, Алматы) заменяется **полной сеткой часовых поясов России, подписанной разницей с Москвой**. Причина: сеткой городов пользователь из непокрытого региона (Омск, Красноярск, Иркутск, Якутск, Магадан, Камчатка) не мог выбрать свой пояс кнопкой и уходил в текстовый ввод; при этом «МСК+N» — привычный жителю России способ назвать свой пояс.

Все 11 часовых поясов России, по два в ряд, город-якорь для узнаваемости (`TIMEZONE_BUTTONS`):

| подпись кнопки | callback | UTC |
|----------------|----------|-----|
| МСК-1 · Калининград | `tz:Europe/Kaliningrad` | +2 |
| МСК · Москва, Санкт-Петербург | `tz:Europe/Moscow` | +3 |
| МСК+1 · Самара | `tz:Europe/Samara` | +4 |
| МСК+2 · Екатеринбург | `tz:Asia/Yekaterinburg` | +5 |
| МСК+3 · Омск | `tz:Asia/Omsk` | +6 |
| МСК+4 · Красноярск | `tz:Asia/Krasnoyarsk` | +7 |
| МСК+5 · Иркутск | `tz:Asia/Irkutsk` | +8 |
| МСК+6 · Якутск | `tz:Asia/Yakutsk` | +9 |
| МСК+7 · Владивосток | `tz:Asia/Vladivostok` | +10 |
| МСК+8 · Магадан | `tz:Asia/Magadan` | +11 |
| МСК+9 · Камчатка | `tz:Asia/Kamchatka` | +12 |
| Другой город — напишу текстом | `tz:other` | — |

Города СНГ (Минск, Алматы и прочие) уходят с кнопок в текстовый фолбэк — раскладка кнопок посвящена России, остальные проходят через «Другой город». Словарь `CITY_TZ` при этом **расширяется** крупными городами России и СНГ, чтобы текстовый ввод попадал с первого раза; города СНГ из прежних кнопок (Минск, Алматы) в словаре обязаны остаться. Часовой пояс, сохраняемый в `users.timezone`, — по-прежнему IANA-идентификатор; подпись «МСК±N» существует только на кнопке.

### Подсказки команд при вводе «/» (добавлено 2026-07-19, решение владельца)

Бот регистрирует список команд через `setMyCommands` (дефолтный scope) — Telegram показывает их при вводе «/». Список: `/start` — «Начать работу с ассистентом», `/help` — «Что умеет ассистент», `/feedback` — «Отправить отзыв разработчику». Админ-команды (`/reject`) не публикуются. Механика — одноразовая команда `python -m gateway set-my-commands` (по образцу set-webhook/set-menu-button), выполняется после деплоя при изменении списка; фиксируется в runbook-deploy.

### 2.5 Капельные подсказки (drip hints)

Одноразовые (на всю жизнь пользователя) сообщения-дописки, отправляются отдельным сообщением после основного ответа.

**`HINT_FINANCE_TEXT`** — после первого успешного вызова любого финансового инструмента:

```
Кстати: все расходы наглядно — на дашборде «Финансы» (кнопка меню). А ещё можно задать бюджет: «поставь бюджет 10 000 на еду в месяц» — предупрежу, когда он будет подходить к концу.
```

**`HINT_REMINDER_TEXT`** — после первого успешного вызова любого reminder-инструмента:

```
Кстати: все напоминания видны в календаре (кнопка меню). Могу и повторяющиеся: «напоминай каждый понедельник в 9 про планёрку».
```

**`HINT_VOICE_TEXT`** — после первой успешно распознанной голосовухи:

```
Кстати: голосовые я понимаю до 5 минут — удобно надиктовывать расходы и напоминания на ходу.
```

Подсказка после первого PDF уже реализована (commit c097946) — не трогать.

## 3. Механика

### 3.0 Отключение ручного одобрения

- Новый (неизвестный) пользователь: создаётся сразу со статусом `active` (вместо `pending`), в той же обработке ставится ключ `onboarding:tz_pending:{tg_user_id}` (существующий TTL 7 дней) и отправляется `START_TIMEZONE_PROMPT_TEXT` с кнопками городов; админам — информационное уведомление.
- Легаси-строки со статусом `pending` (если остались в БД): при следующем контакте пользователя автоматически переводить в `active` и вести по тому же пути выбора города — очередь заявок дренируется сама.
- Статусы `rejected`/`banned` и их обработка сохраняются (модерация остаётся возможной).
- Админ-команды: `/approve` и `/pending` можно удалить либо оставить как no-op-совместимые — решение: **удалить `/approve`, `/pending`**, оставить `/reject` как инструмент модерации (работает и по активным пользователям). Тексты `ADMIN_NEW_APPLICATION_TEXT`, `PENDING_REPEAT_TEXT`, `APPLICATION_RECEIVED_TEXT` удалить; `_admin_usage` обновить.
- Схема БД не меняется (enum статусов остаётся как есть), миграции не нужны.

### 3.1 Два welcome-сообщения

В `worker/onboarding.py` `_save_timezone` отправляет оба сообщения сама через существующий send-колбэк (`self._send`) **последовательно с await** (гарантия порядка: сообщение 1, затем 2) и возвращает `None` вместо текста. Проверить, что возврат `None` из этого пути не порождает пустой ответ (в callback-пути `None` уже означает «ничего не отправлять»). Оба сообщения — через путь с конвертацией markdown → HTML; если текущий `self._send` онбординга шлёт только plain — добавить флаг/использовать метод сендера с `format_as_html=True`.

### 3.2 Drip hints

- Ключ факта показа: `onboarding:hint:{axis}:{tg_user_id}`, axis ∈ {`finance`, `reminder`, `voice`}. Redis **redis-queue** (персистентный, noeviction+AOF — подсказка одноразовая навсегда), `SET NX`, без TTL.
- Триггеры finance/reminder: в воркере после успешного завершения интерактивной задачи (`text`, включая транскрипт голоса; `agent_task` НЕ триггерит) по `AgentResult.tool_names` — использовать те же множества инструментов, что и `mini_app_button_for_tools` в `worker/reply.py` (не дублировать списки — вынести/переиспользовать).
- Триггер voice: в `VoiceProcessor` после успешной транскрипции (тем же send-колбэком, что предупреждения лимитов 5.4).
- Доставка: fire-and-forget отдельным сообщением после основного ответа, по образцу уведомлений 80% (5.4). Ошибка отправки/Redis — только warning в лог, задача не падает (fail-open: при недоступном Redis подсказка просто не отправляется, допустим и дубль).
- За одну задачу — не более одной подсказки каждого типа; если сработали и finance, и reminder — отправить обе (редкий случай, допустимо).

### 3.3 Не входит в скоуп

- Никаких изменений инструментов, tool_calls_log, промптов, Mini App.
- Тексты отказов/лимитов/фидбека не меняются (кроме перечисленных констант).
