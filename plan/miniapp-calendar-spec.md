# Спецификация: экран «Календарь» (Mini App, этап 9.2)

Контекст исполнителю: этот документ самодостаточен вместе с plan/stage-9.md (раздел «Ограничения и принципы» действует целиком) и блоком «Принятые архитектурные решения» мастер-плана. Данные — таблица `scheduled_tasks` (plan/db-schema.md), существующая логика напоминаний — `tools/reminders.py`, шедулер — `worker/scheduler.py`.

## Назначение

Пользователь видит свои напоминания и фоновые задачи на сетке месяца, проваливается в день, создаёт одноразовое напоминание и отменяет существующее — не выходя из Telegram. Это витрина над теми же данными, которыми управляет бот: никакой отдельной «календарной» сущности не появляется.

## UI

### Экран «Месяц» (главный)

- Сетка месяца (пн–вс, локаль ru), заголовок «Июль 2026», стрелки ← → и свайп для перехода по месяцам; кнопка «Сегодня».
- В ячейке дня — до 3 точек-маркеров: синяя = reminder, фиолетовая = agent_task; больше трёх — «+N». Сегодняшний день выделен.
- Тап по дню → список на день (нижняя шторка или отдельный экран — на вкус исполнителя, но с BackButton Telegram).
- Пустой месяц — плейсхолдер: «Пока пусто. Напишите боту "напомни..." или создайте напоминание кнопкой ниже».

### Список на день

- Отсортирован по времени. Элемент: время (HH:MM, таймзона пользователя), текст (payload для reminder, title для agent_task), бейдж «повтор» с человекочитаемым описанием (см. «Описание cron»), для agent_task — иконка робота.
- Прошедшие one-off (status=done) за выбранный день показываются приглушённо — пользователь видит, что напоминание сработало. cancelled не показываются.
- Свайп/кнопка «Отменить» на активном элементе → подтверждение → отмена. agent_task тоже можно отменить (это cancel_scheduled_task).
- Кнопка «+ Напоминание» (Telegram MainButton): форма — текст (до 500 символов), дата (по умолчанию выбранный день), время; только one-off. Повторяющиеся создаются через бота — форма cron в UI v1 не делается (см. «Отложено v2»).

## API (webapp, все — под auth 9.1 и RLS)

### GET /app/api/calendar?from=YYYY-MM-DD&to=YYYY-MM-DD

- Диапазон в таймзоне пользователя, максимум 62 дня (иначе 400) — фронт запрашивает месяц с хвостами недель.
- Выбирает `scheduled_tasks` пользователя: `status='active'` целиком + `status='done'` c `next_run_at` внутри диапазона (для приглушённых прошедших).
- **Развёртка повторов**: для строк с `cron_expr` — croniter в `users.timezone` от начала диапазона, все вхождения до конца диапазона, кап 100 occurrences на задачу (защита от `* * * * *`; при превышении — усечь и пометить `truncated: true`). One-off — единственное вхождение `next_run_at`.
- Ответ: `{ days: { "2026-07-21": [ {id, kind, text, time_local, occurs_at, recurring: bool, repeat_human: str|null, status} ] } }`. `occurs_at` — ISO 8601 с оффсетом таймзоны пользователя.
- Реализация развёртки — чистая функция в `core/` (например `core/calendar_view.py::expand_occurrences`) с юнит-тестами: границы месяца, вхождение ровно на границе диапазона, DST-переход, кап 100.

### POST /app/api/reminders

- Тело: `{ text: str, remind_at_local: "YYYY-MM-DDTHH:MM" }` (интерпретируется в таймзоне пользователя).
- Правила — те же, что у инструмента create_reminder: прошлое время → 422 с текстом «Это время уже прошло», лимит 25 активных → 422. Контракт Mini App дополнительно валидирует текст 1..500 на уровне входной pydantic-модели до вызова общего сервиса. Это API-ограничение не переносится задним числом в `CreateReminderArgs`: историческое поведение LLM-инструмента для уже сформированных аргументов и его существующие error/payload-контракты не меняются.
- **Общая логика**: вынести из tools/reminders.py доменную функцию (например `core/reminders_service.py::create_reminder(user_id, text, remind_at_utc, cron_expr=None)`) — инструмент LLM и API вызывают её для лимита активных и записи; поведение инструмента не меняется (существующие тесты — регресс-контроль). Валидация transport-контрактов остаётся в адаптерах: API — 1..500, инструмент — текущая `CreateReminderArgs`. daily_limit инструмента (30/день) относится к вызовам LLM и на API не распространяется; вместо него действует rate limit 9.1.
- В `tool_calls_log` API-вызовы не пишутся (это лог инструментов LLM); наблюдаемость — структурные логи webapp + метрика `webapp_reminders_created_total`.

### DELETE /app/api/scheduled/{id}

- Отмена (status='cancelled') своей задачи любого kind; уже done/cancelled → 409; чужая/несуществующая под RLS невидима → 404.
- Через ту же доменную функцию, что cancel_reminder/cancel_scheduled_task.

## Описание cron («повтор» человеком)

Хелпер `repeat_human(cron_expr)` для типовых паттернов, которые генерирует create_reminder: ежедневно («каждый день в 09:00»), еженедельно («по понедельникам в 18:30»), ежемесячно («1-го числа в 12:00»); прочее — «по расписанию». Юнит-тесты на все ветки.

## Крайние случаи

- Пользователь сменил таймзону после создания напоминания: развёртка всегда в текущей `users.timezone` — совпадает с поведением шедулера (пересчёт croniter в таймзоне пользователя).
- `next_run_at` в прошлом у active-задачи (шедулер ещё не дотикал) — показывать в её дне как активную.
- Создание напоминания «сегодня через 2 минуты» — допустимо (как у инструмента).
- Двойной тап по «Отменить» / повторный POST — идемпотентность на уровне UI (блокировка кнопки) + 409 от API; ключи idem не нужны (нет переигровки очереди).

## Тесты (без сети, MockLLM не нужен)

- Юнит: развёртка cron (≥6 кейсов), repeat_human, валидация дат/лимитов.
- API: auth-требование на каждом эндпоинте; RLS-изоляция (пользователь A не видит и не может отменить задачу B — 404); создание→появление в GET /calendar; отмена→исчезновение; done в диапазоне виден, cancelled — нет.

## Приёмка (живой e2e)

1. У владельца в боте уже есть напоминания (включая повторяющееся) — все видны на сетке в правильные дни/время МСК.
2. Создать напоминание из Mini App на +5 минут → шедулер доставил в чат → на следующий день оно приглушённое в списке дня.
3. Отменить из Mini App → бот в ответ на «мои напоминания» его не показывает.
4. Запись в progress.md после полной приёмки по регламенту.

## Отложено v2 (не делать без решения владельца)

- Создание/редактирование повторяющихся напоминаний из UI (форма расписания).
- Перенос напоминания drag'n'drop или редактирование времени.
- События Google Calendar как второй источник (после 4.7).
- Недельный/списочный режим отображения.
