# Этап 6. Наблюдаемость и надёжность (4–5 дней, параллельно с бетой)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

6.1. Structured logging (JSON) с trace_id через путь: гейтвей → очередь → воркер → tool-вызовы.

**Детализация 6.1 (2026-07-12).** Без новых зависимостей — JSON-форматтер на stdlib logging.

- **core/logging_setup.py** — `setup_logging(service: str)`: единая инициализация для gateway/worker (шедулер и outbox живут в процессе воркера). Формат по env `LOG_FORMAT`: `json` (дефолт) | `text` (локальная отладка, текущий вид). JSON-строка: `ts` (ISO 8601 UTC), `level`, `logger`, `service`, `trace_id`, `message` + все поля из `extra` записи (нерезервированные атрибуты LogRecord), значения несериализуемых типов — `str()`. Уровень из env `LOG_LEVEL` (дефолт INFO).
- **core/tracing.py** — `trace_id_var: ContextVar[str | None]` + `set_trace_id/reset_trace_id/current_trace_id`. Форматтер берёт trace_id из записи (`extra`), а при отсутствии — из ContextVar; нет ни там, ни там → `"-"`. Существующие вызовы с `extra={"trace_id": ...}` продолжают работать без правок.
- **Установка ContextVar по пути**: gateway — при приёме update (генерация trace_id происходит при создании envelope — установить тот же uuid вокруг обработки); worker — в начале обработки каждого envelope (уже есть `_TraceIdFilter` — заменяется общим механизмом, поведение `trace_id=...` в text-режиме сохраняется); scheduler/outbox — при обработке каждой строки (trace_id из строки/нового uuid — как уже реализовано в их логах); tool-вызовы наследуют ContextVar автоматически (один asyncio-таск).
- **Правки точек входа**: gateway/__main__ (или фабрика приложения) и worker/__main__ вызывают setup_logging; ручные basicConfig/фильтры заменяются. Формат text обязан остаться прежним по содержанию (grep-совместимость e2e-сценариев: `trace_id=... message`).
- **Тесты**: unit — JSON-строка парсится и содержит ts/level/service/trace_id/message; extra-поля попадают в JSON; trace_id: приоритет extra > ContextVar > "-"; ContextVar изолирован между конкурентными asyncio-тасками (два таска — два trace_id без перемешивания); text-режим сохраняет формат; несериализуемое значение в extra не роняет логирование. Integration не нужен. E2e: воркер с LOG_FORMAT=json на живом окружении — text-конверт с инструментом: каждая строка лога валидный JSON, строки задачи (приём, агентный цикл, tool-вызов, ответ) несут один и тот же trace_id, совпадающий с trace_id envelope; шедулер-тик логируется JSON'ом без ошибок.
6.2. Prometheus + Grafana: глубина очередей, латентность p50/p95, ошибки LLM API, стоимость/пользователь/день, кэш-хит prompt caching. Продуктовый минимум там же: DAU/WAU, retention D7, сообщений/пользователь/день, доля голосовых, распределение tool-вызовов (источники — messages, usage, tool_calls_log; только запросы, без нового кода).

**Детализация 6.2 (2026-07-12).** Разделение: технические метрики — Prometheus (инструментация кодом, зависимость `prometheus-client`); продуктовые и per-user стоимость — Grafana-запросы напрямую в Postgres (роль `metrics_ro`, правка db-schema.md 2026-07-12), нового кода для них нет. «Кэш-хит prompt caching» ИЗ СКОУПА ИСКЛЮЧЁН правкой этой детализации: кэширование не включено (pricing-draft §3.6 — апсайд, не заложено), источника данных нет; колонка usage.cached_tokens остаётся на будущее.

- **core/metrics.py** — единый модуль: реестр по умолчанию prometheus_client, объявления метрик и хелперы. Метрики:
  - gateway: `updates_received_total{kind}`, `updates_enqueued_total{queue}`, `updates_dedup_skipped_total`, `updates_rate_limited_total`;
  - worker: `task_duration_seconds{queue}` (histogram, бакеты 0.5–120с — латентность p50/p95 в Grafana через histogram_quantile), `tasks_processed_total{queue,outcome=ok|error|dlq}`, `queue_depth{queue}` (gauge — фоновый сборщик в воркере, каждые 15с XLEN по всем 32 стримам, суммарно по очереди), `llm_requests_total{model,outcome=ok|429|5xx|timeout|other}`, `llm_fallback_total{from,to}`, `llm_cost_rub_total{queue}` (counter, инкремент вместе с add_spend), `stt_requests_total{outcome}`, `tool_calls_total{tool,status}`;
  - экспозиция: gateway — существующий `/metrics` (заглушка 2.1 заменяется на prometheus_client generate_latest); worker — `start_http_server` на `WORKER_METRICS_PORT` (дефолт 9100, `0` = выключено; в тестах выключено).
- **Инструментация** — точечные вызовы в gateway/app.py (приём/enqueue/dedup/rate-limit), worker/app.py (таймер вокруг process + outcome), core/llm_resilient.py (исходы запросов, fallback), core/spend.py или места вызова add_spend (llm_cost_rub_total), worker/voice.py (stt), core/tools.py (tool_calls_total рядом с записью tool_calls_log). Сбой инструментации не должен ронять задачу (метрики — best-effort, но prometheus_client сам не кидает — обёртки не городить).
- **Роль metrics_ro** — в `infra/postgres/init-roles.sql` идемпотентно (как остальные роли): LOGIN, пароль в dev `dev-local-only`, `BYPASSRLS`, GRANT SELECT ON users, messages, usage, tool_calls_log (+ будущие партиции usage/tool_calls_log: `ALTER DEFAULT PRIVILEGES` от migrator). Из приложения не используется.
- **Compose (infra/docker-compose.dev.yml, дефолтный профиль)**: `prometheus` (prom/prometheus, 127.0.0.1:9090, конфиг infra/prometheus/prometheus.yml — scrape gateway `host.docker.internal:8000/metrics` и worker `host.docker.internal:9100`, интервал 15с; gateway/worker в dev бегут на хосте) и `grafana` (grafana/grafana, 127.0.0.1:3000, анонимный доступ Viewer в dev, admin-пароль env GF_SECURITY_ADMIN_PASSWORD дефолт admin). Volumes на данные обоих.
- **Grafana provisioning (infra/grafana/)**: datasources — Prometheus (http://prometheus:9090) и PostgreSQL (host.docker.internal:6432 через PgBouncer, роль metrics_ro, database assistant); дашборды JSON-файлами:
  - «Технический»: глубина очередей (queue_depth), латентность p50/p95 (histogram_quantile по task_duration_seconds), ошибки LLM (rate llm_requests_total по outcome≠ok + llm_fallback_total), стоимость ₽/день (llm_cost_rub_total rate + SQL по usage: сумма за сегодня и топ-10 пользователей по стоимости за день — «стоимость/пользователь/день»), DLQ (tasks_processed_total{outcome="dlq"});
  - «Продуктовый» (SQL в Postgres): DAU/WAU (distinct user_id по messages за день/7 дней), retention D7 (пользователи с первым сообщением N дней назад, писавшие на 7-й день), сообщений/пользователь/день (медиана/среднее по messages role='user'), доля голосовых (meta->>'modality'='voice' к общему числу user-сообщений), распределение tool-вызовов (count по tool_name из tool_calls_log за 7 дней).
- **Тесты**: unit — счётчики/гистограммы инкрементятся в нужных точках (prometheus_client REGISTRY в тестах изолировать CollectorRegistry-инъекцией или собирать значения через generate_latest), gateway /metrics отдаёт exposition-формат с нашими метриками, сборщик queue_depth пишет gauge с fake-redis, WORKER_METRICS_PORT=0 не поднимает сервер. Integration не нужен. E2e (часть DoD этапа): dev-стек + gateway/worker на хосте с метрик-портами → text-конверт через очередь → Prometheus targets оба UP, task_duration_seconds/llm_requests_total/llm_cost_rub_total ненулевые в Prometheus API; Grafana оба дашборда отвечают (API search + render запросов datasource), продуктовые SQL-панели возвращают строки на живых данных.
