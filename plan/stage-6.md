# Этап 6. Наблюдаемость и надёжность (4–5 дней, параллельно с бетой)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

6.1. Structured logging (JSON) с trace_id через путь: гейтвей → очередь → воркер → tool-вызовы.
6.2. Prometheus + Grafana: глубина очередей, латентность p50/p95, ошибки LLM API, стоимость/пользователь/день, кэш-хит prompt caching. Продуктовый минимум там же: DAU/WAU, retention D7, сообщений/пользователь/день, доля голосовых, распределение tool-вызовов (источники — messages, usage, tool_calls_log; только запросы, без нового кода).
6.3. Sentry для исключений gateway/worker/scheduler.
6.4. Алерты: рост q:interactive, доля 429 от OpenRouter, дневная стоимость выше порога, недоступность SearXNG/Postgres/сервиса эмбеддингов.
6.5. Runbook: падение воркеров, переполнение очереди, недоступность LLM API, восстановление из бэкапа.
6.6. Ежемесячная проверка восстановления бэкапа (календарное правило, не код).
6.7. Критерии выхода из беты в публичный запуск — зафиксировать числами до конца беты (ориентиры: retention D7, стоимость/пользователь в рамках модели pricing-draft.md, ноль инцидентов класса «потеря данных» за 2 недели).

[DoD] Дашборд отражает бету в реальном времени; тестовый алерт доходит; восстановление из бэкапа отрепетировано.
