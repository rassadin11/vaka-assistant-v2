# Этап 6. Наблюдаемость и надёжность (4–5 дней, параллельно с бетой)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

6.1. Structured logging (JSON) с trace_id через путь: гейтвей → очередь → воркер → tool-вызовы.

**Детализация 6.1 (2026-07-12).** Без новых зависимостей — JSON-форматтер на stdlib logging.

- **core/logging_setup.py** — `setup_logging(service: str)`: единая инициализация для gateway/worker (шедулер и outbox живут в процессе воркера). Формат по env `LOG_FORMAT`: `json` (дефолт) | `text` (локальная отладка, текущий вид). JSON-строка: `ts` (ISO 8601 UTC), `level`, `logger`, `service`, `trace_id`, `message` + все поля из `extra` записи (нерезервированные атрибуты LogRecord), значения несериализуемых типов — `str()`. Уровень из env `LOG_LEVEL` (дефолт INFO).
- **core/tracing.py** — `trace_id_var: ContextVar[str | None]` + `set_trace_id/reset_trace_id/current_trace_id`. Форматтер берёт trace_id из записи (`extra`), а при отсутствии — из ContextVar; нет ни там, ни там → `"-"`. Существующие вызовы с `extra={"trace_id": ...}` продолжают работать без правок.
- **Установка ContextVar по пути**: gateway — при приёме update (генерация trace_id происходит при создании envelope — установить тот же uuid вокруг обработки); worker — в начале обработки каждого envelope (уже есть `_TraceIdFilter` — заменяется общим механизмом, поведение `trace_id=...` в text-режиме сохраняется); scheduler/outbox — при обработке каждой строки (trace_id из строки/нового uuid — как уже реализовано в их логах); tool-вызовы наследуют ContextVar автоматически (один asyncio-таск).
- **Правки точек входа**: gateway/__main__ (или фабрика приложения) и worker/__main__ вызывают setup_logging; ручные basicConfig/фильтры заменяются. Формат text обязан остаться прежним по содержанию (grep-совместимость e2e-сценариев: `trace_id=... message`).
- **Тесты**: unit — JSON-строка парсится и содержит ts/level/service/trace_id/message; extra-поля попадают в JSON; trace_id: приоритет extra > ContextVar > "-"; ContextVar изолирован между конкурентными asyncio-тасками (два таска — два trace_id без перемешивания); text-режим сохраняет формат; несериализуемое значение в extra не роняет логирование. Integration не нужен. E2e: воркер с LOG_FORMAT=json на живом окружении — text-конверт с инструментом: каждая строка лога валидный JSON, строки задачи (приём, агентный цикл, tool-вызов, ответ) несут один и тот же trace_id, совпадающий с trace_id envelope; шедулер-тик логируется JSON'ом без ошибок.
6.2. Prometheus + Grafana: глубина очередей, латентность p50/p95, ошибки LLM API, стоимость/пользователь/день, кэш-хит prompt caching. Продуктовый минимум там же: DAU/WAU, retention D7, сообщений/пользователь/день, доля голосовых, распределение tool-вызовов (источники — messages, usage, tool_calls_log; только запросы, без нового кода).

**Детализация 6.2 (2026-07-12).** Разделение: технические метрики — Prometheus (инструментация кодом, зависимость `prometheus-client`); продуктовые и per-user стоимость — Grafana-запросы напрямую в Postgres (роль `metrics_ro`, правка db-schema.md 2026-07-12), нового кода для них нет. «Кэш-хит prompt caching» ИЗ СКОУПА ИСКЛЮЧЁН правкой этой детализации: кэширование не включено (pricing-draft §3.6 — апсайд, не заложено), источника данных нет; колонка usage.cached_tokens остаётся на будущее.

- **core/metrics.py** — единый модуль: реестр по умолчанию prometheus_client, объявления метрик и хелперы. Метрики:
  - gateway: `updates_received_total{kind}`, `updates_enqueued_total{queue}`, `updates_dedup_skipped_total`, `updates_rate_limited_total`;
  - worker: `task_duration_seconds{queue}` (histogram, бакеты 0.5–120с — латентность p50/p95 в Grafana через histogram_quantile), `tasks_processed_total{queue,outcome=ok|error|dlq}`, `queue_depth{queue}` (gauge — фоновый сборщик в воркере, каждые 15с по всем 32 стримам, суммарно по очереди; правка 2026-07-20: считать реальный бэклог через `XINFO GROUPS` — `lag` (ещё не доставленные) + `pending` (доставленные, не заACKнутые), а не XLEN: записи после ACK не удаляются, поэтому XLEN — это счётчик всех сообщений за всю историю (до trim MAXLEN 100k) и алерт InteractiveQueueBacklog срабатывал ложно навсегда после 100-го сообщения; `lag=nil` (после XDEL/trim) трактовать как 0, отсутствующий стрим/группу — как 0), `llm_requests_total{model,outcome=ok|429|5xx|timeout|other}`, `llm_fallback_total{from,to}`, `llm_cost_rub_total{queue}` (counter, инкремент вместе с add_spend), `stt_requests_total{outcome}`, `tool_calls_total{tool,status}`;
  - экспозиция: gateway — существующий `/metrics` (заглушка 2.1 заменяется на prometheus_client generate_latest); worker — `start_http_server` на `WORKER_METRICS_PORT` (дефолт 9100, `0` = выключено; в тестах выключено).
- **Инструментация** — точечные вызовы в gateway/app.py (приём/enqueue/dedup/rate-limit), worker/app.py (таймер вокруг process + outcome), core/llm_resilient.py (исходы запросов, fallback), core/spend.py или места вызова add_spend (llm_cost_rub_total), worker/voice.py (stt), core/tools.py (tool_calls_total рядом с записью tool_calls_log). Сбой инструментации не должен ронять задачу (метрики — best-effort, но prometheus_client сам не кидает — обёртки не городить).
- **Роль metrics_ro** — в `infra/postgres/init-roles.sql` идемпотентно (как остальные роли): LOGIN, пароль в dev `dev-local-only`, `BYPASSRLS`, GRANT SELECT ON users, messages, usage, tool_calls_log (+ будущие партиции usage/tool_calls_log: `ALTER DEFAULT PRIVILEGES` от migrator). Из приложения не используется.
- **Compose (infra/docker-compose.dev.yml, дефолтный профиль)**: `prometheus` (prom/prometheus, 127.0.0.1:9090, конфиг infra/prometheus/prometheus.yml — scrape gateway `host.docker.internal:8000/metrics` и worker `host.docker.internal:9100`, интервал 15с; gateway/worker в dev бегут на хосте) и `grafana` (grafana/grafana, 127.0.0.1:3000, анонимный доступ Viewer в dev, admin-пароль env GF_SECURITY_ADMIN_PASSWORD дефолт admin). Volumes на данные обоих.
- **Grafana provisioning (infra/grafana/)**: datasources — Prometheus (http://prometheus:9090) и PostgreSQL (host.docker.internal:6432 через PgBouncer, роль metrics_ro, database assistant); дашборды JSON-файлами:
  - «Технический»: глубина очередей (queue_depth), латентность p50/p95 (histogram_quantile по task_duration_seconds), ошибки LLM (rate llm_requests_total по outcome≠ok + llm_fallback_total), стоимость ₽/день (llm_cost_rub_total rate + SQL по usage: сумма за сегодня и топ-10 пользователей по стоимости за день — «стоимость/пользователь/день»), DLQ (tasks_processed_total{outcome="dlq"});
  - «Продуктовый» (SQL в Postgres): DAU/WAU (distinct user_id по messages за день/7 дней), retention D7 (пользователи с первым сообщением N дней назад, писавшие на 7-й день), сообщений/пользователь/день (медиана/среднее по messages role='user'), доля голосовых (meta->>'modality'='voice' к общему числу user-сообщений), распределение tool-вызовов (count по tool_name из tool_calls_log за 7 дней).
- **Тесты**: unit — счётчики/гистограммы инкрементятся в нужных точках (prometheus_client REGISTRY в тестах изолировать CollectorRegistry-инъекцией или собирать значения через generate_latest), gateway /metrics отдаёт exposition-формат с нашими метриками, сборщик queue_depth пишет gauge с fake-redis, WORKER_METRICS_PORT=0 не поднимает сервер. Integration не нужен. E2e (часть DoD этапа): dev-стек + gateway/worker на хосте с метрик-портами → text-конверт через очередь → Prometheus targets оба UP, task_duration_seconds/llm_requests_total/llm_cost_rub_total ненулевые в Prometheus API; Grafana оба дашборда отвечают (API search + render запросов datasource), продуктовые SQL-панели возвращают строки на живых данных.

6.4. Алерты: рост q:interactive, доля 429 от OpenRouter, дневная стоимость выше порога, недоступность SearXNG/Postgres/сервиса эмбеддингов.

**Детализация 6.4 (2026-07-12).** Решение: алертинг — **Grafana Unified Alerting** (Grafana уже в стеке 6.2), отдельный Alertmanager не вводим — меньше контейнеров и нет проблемы прокидывания секретов в его конфиг (Grafana provisioning умеет `$VAR` из env). Нового Python-кода нет — только infra-конфиги.

- **Проверка доступности сервисов** — `blackbox_exporter` (prom/blackbox-exporter, дефолтный профиль compose, порт наружу не нужен): модуль `http_2xx` для SearXNG (`http://searxng:8080/healthz`) и сервиса эмбеддингов (`http://embeddings:8000/healthz`), модуль `tcp_connect` для Postgres (`postgres:5432`). В prometheus.yml — job `blackbox` со стандартным relabel (`__param_target` → `instance`, адрес → `blackbox:9115`). Нюанс dev: embeddings живёт в профиле `ml` — при выключенном профиле probe честно падает и алерт активен; это допустимо (severity warning), задокументировать в infra/README.md.
- **Правила** (Grafana alert rules provisioning-файлом `infra/grafana/provisioning/alerting/rules.yml`, datasource Prometheus, группа `assistant`, интервал оценки 1m; пороги — константы в файле, менять правкой файла):
  1. `InteractiveQueueBacklog` (warning): `sum(queue_depth{queue="interactive"}) > 100` в течение 5m — воркеры не успевают или лежат.
  2. `LLM429RatioHigh` (warning): доля 429 `sum(rate(llm_requests_total{outcome="429"}[10m])) / sum(rate(llm_requests_total[10m])) > 0.2` в течение 10m, с гвардом на минимальный трафик `sum(rate(llm_requests_total[10m])) > 0.05` (иначе шум на единичных запросах).
  3. `DailyLLMCostHigh` (critical): `sum(increase(llm_cost_rub_total[24h])) > 500` (₽/сутки; ориентир — 2× ожидаемая дневная стоимость закрытой беты по pricing-draft, финальный порог пересмотреть перед 7.4).
  4. `TargetDown` (critical): `up == 0` в течение 3m — gateway или worker не отдают метрики.
  5. `ServiceProbeFailed`: `probe_success == 0` в течение 3m — по одному правилу-инстансу на цель через label; Postgres/SearXNG — critical, embeddings — warning (см. нюанс профиля ml).
- **Доставка** — contact point `telegram-admin` (provisioning-файл `contact-points.yml`): тип telegram, `bottoken: $TELEGRAM_ALERT_BOT_TOKEN`, `chatid: $TELEGRAM_ALERT_CHAT_ID` — значения приходят в контейнер Grafana через compose `environment: ${TELEGRAM_ALERT_BOT_TOKEN:-}` (секрет в репо не попадает; в dev экспортировать перед `make dev-up`, боевой токен бота годится). Notification policy по умолчанию → `telegram-admin`, тексты уведомлений — краткий русский шаблон (имя алерта, состояние firing/resolved, значение). Без заданного токена Grafana стартует, алерты видны в UI, доставка молча не работает — приемлемо для dev, отметить в README.
- **Тесты**: юнитов нет (Python-код не меняется); существующий CI-прогон должен остаться зелёным (compose-конфиги валидируются `docker compose config` в e2e). E2e: dev-стек + gateway/worker на хосте с env-токенами → все blackbox-цели `probe_success == 1`; остановить searxng → в Grafana API правило `ServiceProbeFailed` переходит в Firing и в админский Telegram приходит уведомление; запустить обратно → приходит resolved; тестовое сообщение contact point через Grafana API (`/api/v1/provisioning/contact-points` + test) доходит.
6.5. Runbook: падение воркеров, переполнение очереди, недоступность LLM API, восстановление из бэкапа.

**Детализация 6.5 (2026-07-12).** Runbook — `docs/runbook.md`, русский, для оператора (сегодня это владелец). Кода нет — только документ; пишет Claude (текстовый документ, делегирование дороже). Принципы: каждый сценарий в формате «симптом/алерт → диагностика (конкретные команды) → действия → проверка, что починилось»; команды — реальные и проверенные (Makefile, docker compose, redis-cli, Grafana/Prometheus API), никаких гипотетических; сценарии привязаны к алертам 6.4 по именам правил. Состав:

1. **Шапка**: где смотреть (Grafana localhost:3000, Prometheus localhost:9090, алерты в Telegram при заданном токене), где логи (docker logs, JSON-формат 6.1, поиск по trace_id).
2. **Падение воркеров** (`TargetDown` worker): диагностика (процесс на хосте, docker для будущего prod, логи последних ошибок), рестарт, проверка XPENDING/redelivery — задачи не теряются (at-least-once 2.5), контроль разгребания очереди.
3. **Переполнение interactive-очереди** (`InteractiveQueueBacklog`): диагностика XLEN/XPENDING по партициям, причины (воркеры лежат/зависли на локе, наплыв), действия (рестарт воркеров, доп. воркеры, режим interactive-only 2.6), обработка DLQ `q:dlq` — осмотр XRANGE, ручная переигровка или списание.
4. **Недоступность LLM API** (`LLM429RatioHigh`, алерты circuit breaker 3.5 в Telegram): что происходит автоматически (retry ×3, fallback-модель, cooldown 300с) и когда вмешиваться (оба провайдера лежат, ключ протух, лимит OpenRouter), проверка ключа/лимитов, семафор sem:openrouter при подозрении на утечку слотов.
5. **Недоступность сервисов** (`ServiceProbeFailed`/`EmbeddingsProbeFailed`): рестарт контейнера, особенности (searxng — деградация web_search с понятным отказом; embeddings — профиль ml, деградация памяти/поиска документов без падения задач; postgres — критично, см. п.6).
6. **Восстановление из бэкапа**: pgBackRest (repo1 7d / repo2 4w + WAL), `make backup`/`backup-weekly`/`backup-check`, репетиция восстановления `make restore-to-dev` (маркерная проверка PASS), ловушка `--profile restore down` из infra/README; восстановление Redis-queue из RDB-снапшота; дамп Infisical.
7. **Общий сбор контекста при инциденте**: trace_id из ответа/логов → grep по логам всех сервисов, tool_calls_log/messages в БД под service-ролью.

Prod-специфика (systemd/Ansible, боевые хосты) — дописывается в 1B; сейчас команды dev-стека с пометками. **DoD 6.5**: документ покрывает все 4 сценария из плана + сервисы + DLQ; каждая команда прогнана вживую при написании (или явно помечена «prod, появится в 1B»); ссылка на runbook из infra/README.md.
6.6. Ежемесячная проверка восстановления бэкапа (календарное правило, не код).
6.7. Критерии выхода из беты в публичный запуск — зафиксировать числами до конца беты (ориентиры: retention D7, стоимость/пользователь в рамках модели pricing-draft.md, ноль инцидентов класса «потеря данных» за 2 недели).

**Черновик критериев 6.7 (2026-07-14, ЧИСЛА НЕ ФИНАЛЬНЫЕ — на подтверждении у владельца, вопрос в Telegram).** Выход из закрытой беты в публичный запуск разрешён, когда одновременно выполнены:

1. **Retention**: D7 ≥ 30% — доля активированных бета-пользователей (прошли онбординг и написали ≥3 сообщений), которые вернулись к боту на 7-й день ±1 (продуктовый дашборд 6.2, messages).
2. **Экономика**: медианная себестоимость активного пользователя ≤ 140₽/мес и средняя стоимость сообщения ≤ 0.5₽ — в рамках модели pricing-draft §2/§5 (маржа Базового ≥ ~36%); считается по usage за последние 30 дней.
3. **Надёжность**: ноль инцидентов класса «потеря данных» за последние 14 дней беты; доля сообщений, ушедших в DLQ, < 0.5% за те же 14 дней.
4. **Готовность обвязки** (не метрика, чек-лист): закрыт этап 7 в части оферты/Tribute/триала; критические алерты 6.4 подключены к боевому Telegram; ежемесячная проверка бэкапа (6.6) прошла хотя бы один раз на проде.

Минимальный размер выборки для п.1–2: ≥ 20 активированных пользователей; меньше — бету продлеваем, по числам не судим.
