# Этап 8. Подготовка к росту (по порогам, не по календарю)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

Триггеры и действия:

8.1. Постоянная загрузка CPU/RAM БД > 60% → вынос Postgres на отдельный узел; read-реплика для аналитики.
8.2. Ручное управление числом воркеров стало рутиной → k3s (2–3 узла) + KEDA-автоскейл по глубине очереди; zero-downtime деплой.
8.3. Redis Streams упирается в пропускную способность или нужна геораспределённость → миграция очереди на NATS JetStream (интерфейс очереди изолирован в 2.3, воркеры не переписываются).
8.4. Выбор моделей пользователями → активация роутера из 3.6 + тариф Pro из 7.4.
8.5. Сервис эмбеддингов упирается в CPU (латентность автоинжекта растёт) → второй экземпляр за балансировкой / вынос на отдельный узел; сервис stateless, воркеры не меняются. Крайний вариант — семантический поиск по messages (если суммаризация станет болью пользователей) — потребует эмбеддинги на каждое сообщение и пересмотра мощностей.

[DoD] Не применимо; этап событийный.
