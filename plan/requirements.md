# Документ требований (v1) — пункт 0.1

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md). Технические решения и стек — там; здесь — продуктовые требования, границы v1 и политика данных.

## 1. Продукт

Персональный ассистент в Telegram: один бот, диалоговый интерфейс (текст + голосовые), выполняет задачи пользователя через инструменты (LLM с function calling). Multi-tenant: общая инфраструктура, строгая изоляция данных пользователей (PostgreSQL RLS).

Целевая аудитория v1: русскоязычные пользователи Telegram, персональное использование (не команды). Только приватные чаты — группы вне scope v1.

## 2. Ключевые сценарии (v1)

Спецификация инструментов — [../tools_registry_v1.md](../tools_registry_v1.md); здесь пользовательская формулировка.

1. **Финансы** — учёт расходов/доходов свободным текстом («потратил 700 на такси»), категоризация, запросы статистики за период, графики (PNG), бюджеты и предупреждения о превышении.
2. **Напоминания** — разовые и повторяющиеся, естественным языком, в таймзоне пользователя; доставка в Telegram точно в срок.
3. **Google Calendar + Gmail (отправка)** — просмотр/создание событий календаря, отправка писем от имени пользователя. Подключение через OAuth (scopes: calendar.events, gmail.send), мутации внешних систем — только после явного подтверждения пользователем.
4. **Веб-поиск** — поиск в интернете (self-hosted SearXNG) и чтение страниц; ответы с указанием источников.
5. **Документы (PDF)** — загрузка PDF, извлечение текста, семантический поиск по содержимому, вопросы по документам.

Дополнительно (кросс-сценарное): долговременная память о пользователе (факты, предпочтения), голосовой ввод (STT → обычный текстовый путь), фоновые задачи по расписанию (этап 5).

## 3. Монетизация

- Подписка ~249 ₽/мес через Tribute (webhook-активация). Тарифная сетка и юнит-экономика — [pricing-draft.md](pricing-draft.md) (финализация перед 7.4).
- Триал: 7 дней ИЛИ 50 сообщений (что раньше). Grace-период после истечения подписки.
- Лимиты: дневные бюджеты в ₽/день на LLM-расходы, счётчики тарифных лимитов, уведомление на 80%.
- Закрытая бета до публичного запуска: ручное одобрение пользователей, оплата отключена.

## 4. Политика данных (GDPR-минимум)

- **Экспорт**: пользователь получает все свои данные по запросу (команда в боте, этап 7.5).
- **Удаление**: полное удаление аккаунта и данных по запросу; каскад по всем 13 таблицам.
- **Сроки хранения**: автоудаление по категориям данных — параметры фиксируются в обсуждении №9 (перед 7.5).
- **LLM-провайдеры**: только no-training (OpenRouter `data_collection: deny`); передача данных LLM- и STT-провайдерам раскрывается в privacy policy.
- **Google Limited Use**: данные Workspace API не передаются для обучения моделей и не используются вне заявленных сценариев.
- **Секреты пользователей**: OAuth-токены — envelope-шифрование в Postgres; пароли не принимаются в принципе.
- **152-ФЗ**: применимость и выбор юрисдикции хостинга — обсуждение №3 (перед 1B).
- Privacy policy публикуется на домене до подачи заявки на верификацию Google (0.6).

## 5. Нефункциональные требования

- Изоляция: данные пользователя недоступны из контекста другого (RLS, тест в CI).
- Отклик: подтверждение получения (typing/промежуточные сообщения) сразу; гейтвей отвечает Telegram < 100 мс.
- Надёжность: сообщения не теряются (очередь с redelivery + DLQ); mutating-действия идемпотентны при переигровке.
- Стоимость: расход на пользователя ограничен дневным бюджетом; деградация вместо отказа при исчерпании.
- Тесты не ходят в сеть и не тратят токены (Mock-провайдеры).
