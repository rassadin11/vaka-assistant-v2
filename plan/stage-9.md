# Этап 9. Mini App: календарь + финансовый дашборд

Источник: бета-фидбек владельца 2026-07-17 (docs/bot-analysis-2026-07-17.md, п.4 — календарь был осознанно исключён из доработок под отдельный мастер-план) + решение владельца 2026-07-17 расширить тему финансов до визуального дашборда.

Назначение: визуальная поверхность поверх бота — Telegram Mini App (WebApp) на существующем домене. Две фичи v1: **календарь** (напоминания и фоновые задачи на сетке месяца) и **финансовый дашборд** (список трат, графики, разбивка по категориям, выбор периода, мини-резюме от нейросети). Детальные спецификации экранов — отдельные документы:

- [miniapp-calendar-spec.md](miniapp-calendar-spec.md) — календарь;
- [miniapp-finance-spec.md](miniapp-finance-spec.md) — финансы.

Архитектурная подготовка и передача исполнителю:

- [stage-9-architecture.md](stage-9-architecture.md) — принятые границы сервисов, auth/RLS, API, frontend, cache и deployment;
- [stage-9-handoff.md](stage-9-handoff.md) — карта текущего репозитория, батчи реализации и e2e-матрица.

Порядок реализации: 9.1 → 9.2 → 9.3 → 9.4. Экраны 9.2 и 9.3 независимы друг от друга (после 9.1 можно делать в любом порядке; при желании — параллельно в worktree).

## Ограничения и принципы (действуют на весь этап)

- **Инварианты кода из CLAUDE.md действуют полностью**: RLS через `SET LOCAL app.user_id` на каждую транзакцию, stateless-сервисы, секреты из Infisical, тесты без сети.
- **Гейтвей не трогаем** (кроме кнопки меню/клавиатур): архитектурное решение «гейтвей тонкий, без Postgres» сохраняется. Mini App обслуживает новый сервис `webapp/`.
- **Новых таблиц нет.** Календарь читает `scheduled_tasks`, финансы — `transactions` + `budgets`. Если по ходу реализации понадобится новая таблица/колонка — стоп, правка db-schema.md, миграция expand-contract, потом код.
- **Один код для LLM-инструментов и API.** Валидация и запись напоминаний, агрегаты трат — общая логика в `core/` (рефакторинг существующих tools/, поведение инструментов не меняется — регресс ловится существующими тестами).
- **Никакого доверия фронту.** `user_id` берётся только из проверенного initData → сессионного токена; из тела запроса — никогда (тот же принцип, что «user_id не приходит из аргументов LLM»).
- **Языки**: код/идентификаторы — английский, тексты интерфейса — русский.
- Время: API отдаёт ISO 8601 в таймзоне пользователя (`users.timezone`), как для модели; в БД — timestamptz UTC.

## 9.1 Платформа Mini App (фундамент, без экранов)

Состав:

1. **Сервис `webapp/`** — FastAPI-приложение (отдельный контейнер в compose dev+prod): отдаёт собранную статику фронта и JSON API под `/app/api/*`. Healthcheck `/app/healthz`, метрики `/app/metrics` (счётчики запросов, латентность, ошибки auth — стиль 6.2). Structured logging + trace_id (6.1) на каждый запрос.
2. **Auth**: эндпоинт `POST /app/api/auth` принимает `initData` из Telegram.WebApp; валидация HMAC-SHA256 по спецификации Telegram (secret = HMAC("WebAppData", bot_token)), проверка `auth_date` (не старше 1 часа), резолв `tg_user_id → users.id` через узкую `SECURITY DEFINER`-функцию из `db-schema.md` без выдачи webapp роли `service` (только `status=active`; pending/rejected/banned → 403 с человекочитаемым текстом). Ответ — короткоживущий подписанный токен itsdangerous, TTL 12 ч, ключ из Infisical, внутри — только версия и `users.id`; актуальные status/timezone/plan перечитываются под RLS на каждом API-запросе. Все остальные эндпоинты требуют токен (заголовок Authorization), при истечении фронт один раз повторяет auth по initData; mutating-запрос автоматически не переигрывается. Детали — `stage-9-architecture.md`.
3. **RLS-доступ**: webapp подключается ролью `app` через PgBouncer, каждый запрос — `user_transaction(user_id)` из `core/db.py`. Роль `service` в webapp запрещена.
4. **Rate limit**: per-user на API (redis-cache, token bucket по образцу core/rate_limit.py, порядка 60 запросов/мин) — Mini App не должен позволить выкачивать БД в цикле.
5. **Фронт-каркас `miniapp/`**: Vite + Preact + TypeScript, без тяжёлых UI-фреймворков. Подключение `telegram-web-app.js` (официальный скрипт Telegram — единственная внешняя загрузка), тема из `Telegram.WebApp.themeParams` (светлая/тёмная), BackButton, haptic по вкусу. Сборка — стадия node в Docker multi-stage, готовая статика копируется в образ webapp; в CI — job сборки фронта (lint + build).
6. **Маршрутизация и вход**: Caddy — маршрут `/app/*` → webapp (правка infra + Ansible-роль app по образцу 1.11). Вход пользователя: кнопка меню бота (`setChatMenuButton` → web_app URL) + позже inline-кнопки из ответов бота (9.4).
7. **Безопасность**: CSP (self + telegram.org), `X-Frame-Options` не ставить (Mini App живёт в iframe/webview Telegram — проверить, что security-заголовки Caddy из 1.11.2 не ломают загрузку), никаких секретов и user_id в URL.

DoD 9.1: контейнер webapp в dev и prod; `/app` открывается из Telegram у активного пользователя (владелец, живой e2e), у не-активного — отказ; невалидный/просроченный initData → 401 (юнит-тесты на все ветки валидации, включая подделку hash); запрос под RLS возвращает только свои данные (тест с двумя пользователями); rate limit срабатывает; CI собирает фронт и образ.

## 9.2 Экран «Календарь»

Спецификация — [miniapp-calendar-spec.md](miniapp-calendar-spec.md). Кратко: сетка месяца + список на день из `scheduled_tasks` (reminder + agent_task), разворачивание cron-повторов в occurrences на видимый диапазон (croniter в таймзоне пользователя), создание одноразового напоминания и отмена — через общую доменную логику с tools/reminders.py (лимит 25 активных и прочие правила — одни на всех).

DoD 9.2: владелец видит свои напоминания на сетке месяца, повторяющиеся отображаются на каждую дату вхождения; созданное из Mini App напоминание срабатывает через шедулер (живой e2e); отмена из Mini App отменяет и для бота (list_reminders её не показывает); тесты на развёртку cron (границы месяца, таймзона, DST).

## 9.3 Экран «Финансы»

Спецификация — [miniapp-finance-spec.md](miniapp-finance-spec.md). Кратко: выбор периода (день/неделя/месяц/произвольный диапазон), сводка (итог расходов/доходов, сравнение с прошлым периодом), donut по категориям, столбики по дням, прогресс бюджетов (budgets), постраничный список транзакций, удаление транзакции, мини-резюме от нейросети (LLM по агрегатам, кэш, учёт в usage и дневном бюджете ₽).

DoD 9.3: владелец на живых данных видит корректные суммы (сверка с query_transactions), графики соответствуют категориям и дням, произвольный диапазон работает; AI-резюме генерируется, кэшируется (повторное открытие — без нового вызова LLM), записано в usage как background и учтено в spend 5.2; при ≥100% дневного бюджета резюме не генерируется (плашка); удаление транзакции отражается и в ответах бота.

## 9.4 Интеграция с ботом и полировка

Механизм связки бота и Mini App (контракт ответа, сопоставление инструмент→экран, тип
кнопки, конфиг и метрика) зафиксирован в [stage-9-architecture.md](stage-9-architecture.md),
раздел «9.4 Интеграция с ботом». Ниже — состав пункта.

1. Кнопка меню бота на Mini App (`setChatMenuButton` → `MenuButtonWebApp` на `{PUBLIC_URL}/app/`),
   отдельная идемпотентная команда gateway `set-menu-button` по образцу `set-webhook`;
   срабатывает и в проде через deploy-скрипт.
2. Inline-кнопки в ответах бота: после ответа, где сработал инструмент трат/бюджета —
   «Открыть финансы», после ответа с напоминаниями — «Открыть календарь». Кнопка — Telegram
   `web_app`-кнопка на `{PUBLIC_URL}/app/?screen=<finance|calendar>` (открывает нужный экран
   без deep-link зависимости от BotFather). Экран — единственный параметр входа; период и любые
   финансовые данные в URL не передаются (выбор периода остаётся внутри экрана). Сигнал «какой
   инструмент сработал» приходит из `AgentResult` (список имён инструментов), не из промпта.
3. Тексты /help и welcome: упомянуть Mini App (наглядные экраны календаря и трат), без обещаний
   функций v2.
4. Метрики продуктовые: открытие Mini App (счётчик успешной auth) и использование экранов
   (`webapp_requests_total` по route календарь/финансы) — панели в Grafana «Продуктовый» на
   Prometheus-датасорсе. Без user-id-лейблов.
5. Prompt-правка НЕ нужна: инструменты LLM не меняются; текст Mini App в системный промпт агента
   не добавляется.

DoD 9.4: вход в Mini App в один тап из меню и из inline-кнопок; кнопка открывает нужный экран;
неизвестный `screen`/`start_param` безопасно ведёт на экран по умолчанию; /help и welcome
актуальны и не обещают v2; продуктовые метрики видны без user-id-лейблов. Живой e2e (Telegram
Desktop + мобильный, светлая/тёмная тема) — на activation gate.

## DoD этапа

- Живой e2e владельцем на проде: открыть календарь, создать и увидеть напоминание; открыть финансы, посмотреть месяц/произвольный период, прочитать AI-резюме.
- Изоляция: тест «пользователь A не видит данных B» на уровне API (RLS + auth).
- Полная приёмка по регламенту (ruff/mypy/pytest/gitleaks/CI) на каждый пункт; для новых эндпоинтов — записи в логах со trace_id.
- Нагрузка не требуется (бета ≤20 пользователей), но rate limit проверен.

## Риски и открытые вопросы

- **Заголовки Caddy vs webview Telegram** — проверить на 9.1 первым живым запуском (частая грабля Mini App: X-Frame-Options/CSP ломают iframe).
- **Категория investments** (п.11 bot-analysis): enum категорий зафиксирован схемой v1; дашборд её не добавляет. Если владелец захочет — отдельная правка db-schema.md + миграция, не в этом этапе.
- **Редактирование транзакций** (не только удаление) — отложено, см. «Отложено v2» в miniapp-finance-spec.md.
- **Google Calendar** (4.7) сюда не входит: календарь Mini App показывает данные бота; интеграция Google остаётся за 4.7/№5 и при появлении просто добавит источник событий.
