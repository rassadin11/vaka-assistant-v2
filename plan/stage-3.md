# Этап 3. Агентное ядро (5–8 дней)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

3.1. Интерфейс LLMProvider (generate с tools, счётчики токенов/стоимости) + реализация OpenRouterProvider (openai SDK, base_url OpenRouter, модель deepseek) + MockLLMProvider для тестов: сценарные ответы (заданная последовательность tool_calls → финальный ответ), запись входящих вызовов для ассертов. Тесты этапов 3–4 не ходят в сеть и не тратят токены. Роутинг провайдеров: явный allowlist no-training провайдеров в конфиге (параметр order/only OpenRouter) — разные хостеры гоняют разные кванты модели, поведение фиксируется списком, а не категорией.
3.2. Агентный цикл: messages → tool_calls → диспетчер → tool results → повтор до финального ответа. Ограничители: 10 tool-вызовов/задачу, таймаут 120 с, бюджет стоимости/задачу (₽ — деноминация как в 5.2). Задача дольше ~15 с — промежуточное сообщение пользователю по имени исполняемого инструмента («ищу в интернете…», «читаю документ…»); короткие покрывает «typing…» из 2.10.
3.3. Обработка malformed tool calls (специфика DeepSeek): ошибка в контекст, максимум 2 ретрая.
3.4. Контекст-менеджер: сборка контекста строго по [system-prompt-spec.md](system-prompt-spec.md) — блоки A–F, токен-бюджеты из спецификации проверяются кодом. Фоновая суммаризация при превышении бюджета хвоста. Текст системного промпта пишется по той же спецификации; приёмка — eval-набор ≥ 18/20 (раздел 4 спецификации); PROMPT_VERSION логируется в meta сообщений. Eval-прогон — на провайдерах из allowlist 3.1: приёмка на одном хостере не переносится на другой (разные кванты).
3.5. Глобальный семафор конкурентных запросов к OpenRouter (Redis) + retry с экспоненциальной задержкой на 429/5xx. Конфиг fallback-модели той же семьи (напр. deepseek-chat ↔ V4 Flash): переключение после N подряд неудач по всем провайдерам основной, с алертом админу.
3.6. Роутинг моделей: v1 — одна модель (deepseek-chat); интерфейс роутера заложить (вход: тип задачи, план пользователя → модель + множитель бюджета). [Обсудить: правила роутинга при добавлении выбора моделей]
3.7. Персистентность диалога: запись сообщений и ответов в messages в конце задачи; суммаризации — в dialog_summaries (схема — [db-schema.md](db-schema.md)).
3.8. Учёт стоимости: токены, cached_tokens и $ per-запрос — в таблицу usage (схема — [db-schema.md](db-schema.md)).

[DoD] Диалог с памятью работает end-to-end с одним тестовым инструментом; стоимость каждой задачи видна в БД; обрыв воркера не теряет задачу (переигрывается из stream).
