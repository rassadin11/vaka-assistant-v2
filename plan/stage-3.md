# Этап 3. Агентное ядро (5–8 дней)

Часть мастер-плана: [../implementation_plan_v1.md](../implementation_plan_v1.md) — там архитектурные решения, сводный график и регламент передачи задач.

3.1. Интерфейс LLMProvider (generate с tools, счётчики токенов/стоимости) + реализация OpenRouterProvider (openai SDK, base_url OpenRouter, модель deepseek) + MockLLMProvider для тестов: сценарные ответы (заданная последовательность tool_calls → финальный ответ), запись входящих вызовов для ассертов. Тесты этапов 3–4 не ходят в сеть и не тратят токены. Роутинг провайдеров: явный allowlist no-training провайдеров в конфиге (параметр order/only OpenRouter) — разные хостеры гоняют разные кванты модели, поведение фиксируется списком, а не категорией.
3.2. Агентный цикл: messages → tool_calls → диспетчер → tool results → повтор до финального ответа. Ограничители: 10 tool-вызовов/задачу, таймаут 120 с, бюджет стоимости/задачу (₽ — деноминация как в 5.2). Задача дольше ~15 с — промежуточное сообщение пользователю по имени исполняемого инструмента («ищу в интернете…», «читаю документ…»); короткие покрывает «typing…» из 2.10. Наследие этапа 2 (ревизия 2026-07-10, обязательно к реализации здесь): (а) продлевать per-user lock в течение цикла (extend_user_lock из core/locks.py, интервал ~60 с); (б) починить дрейф delivery count — при задаче в 120 с следующее сообщение пользователя копит доставки через reclaim-цикл (см. уточнение в stage-2.md 2.4) и может уехать в DLQ без единой попытки обработки; решение: DLQ-порог считать по счётчику фактических попыток process() (инкремент в момент старта обработки, хранение в envelope.attempt при переигровке), а XPENDING delivery count использовать только как сигнал reclaim'а; (в) TaskContext.user_id (внутренний uuid users.id) брать из резолва, который уже выполняет OnboardingProcessor, — прокинуть в контекст задачи, а не резолвить повторно.
3.3. Обработка malformed tool calls (специфика DeepSeek): ошибка в контекст, максимум 2 ретрая. Детализация (2026-07-10):
   - Malformed — это: (а) вызов инструмента, которого нет в `definitions()` (в т.ч. пустое имя); (б) `arguments_json` не парсится как JSON или парсится не в объект. Валидация — в диспетчере до вызова handler'а: `StaticToolDispatcher.dispatch` парсит `arguments_json` и бросает `MalformedToolCallError(ToolDispatchError)` с человекочитаемой причиной. Валидация аргументов по JSON-схеме инструмента — не здесь, а в реестре 4.1 (реестр §1); механизм ошибок при этом переиспользуется.
   - Обработка в агентном цикле: на malformed-вызов в контекст добавляется tool-сообщение с текстом ошибки (по-русски, с перечислением доступных инструментов при неизвестном имени) — модель получает шанс исправиться. Валидные вызовы из того же ответа исполняются как обычно.
   - Лимит: счётчик malformed-раундов на задачу (раунд = ответ модели, содержащий ≥1 malformed-вызов). После 2 ретраев (3-й malformed-раунд подряд или суммарно за задачу — считаем суммарно, проще и безопаснее) цикл останавливается: `stop_reason="malformed"`, пользователю — заготовленный текст-фолбэк. Лимиты 3.2 (10 tool-вызовов, таймаут, бюджет) продолжают действовать независимо; malformed-вызовы входят в общий счётчик tool-вызовов.
   - Прочие ошибки диспетчера (`ToolDispatchError`, не malformed) идут в контекст как раньше, без отдельного счётчика — политика ретраев исполнения инструментов задаётся реестром в этапе 4.
   - Тесты: сценарии MockLLMProvider — неизвестный инструмент → исправление; битый JSON → исправление; 3 malformed-раунда → фолбэк и `stop_reason="malformed"`; смешанный ответ (валидный + malformed) исполняет валидный.
3.4. Контекст-менеджер: сборка контекста строго по [system-prompt-spec.md](system-prompt-spec.md) — блоки A–F, токен-бюджеты из спецификации проверяются кодом. Фоновая суммаризация при превышении бюджета хвоста. Текст системного промпта пишется по той же спецификации; приёмка — eval-набор ≥ 18/20 (раздел 4 спецификации); PROMPT_VERSION логируется в meta сообщений. Eval-прогон — на провайдерах из allowlist 3.1: приёмка на одном хостере не переносится на другой (разные кванты). Детализация (2026-07-10):
   - **Подсчёт токенов**: tiktoken `cl100k_base` как детерминированное приближение (точный токенизатор DeepSeek не нужен — бюджеты по построению приближённые, запас ×1.5 заложен в pricing-draft). Счётчик — одна функция в core (`count_tokens`), она же считает `messages.tokens` при записи (3.7). Тесты без сети: tiktoken работает офлайн (энкодинг в пакете).
   - **Границы компонента**: `core/context_manager.py` — чистая сборка: вход (факты, последняя суммаризация, хвост messages, профиль динамики C) → system-message A–E с разделителями секций + список F; проверка бюджетов каждого блока кодом (превышение хвоста → сигнал «нужна суммаризация», превышение D/E — усечение с конца, превышение A — ошибка конфигурации). Чтение из БД (messages, dialog_summaries) и запись — в 3.7; в 3.4 входы передаются явно.
   - **Блок D**: до появления memory-инструментов (этап 5) вход фактов — пустой список; сборщик поддерживает интерфейс сразу.
   - **Суммаризация**: функция `summarize_tail` через LLMProvider (промпт суммаризации — часть 3.4, бюджет E ≤ 800 проверяется на результате); триггер и фоновый запуск — в 3.7 вместе с персистентностью.
   - **Промпт**: статичное ядро A — английский, отдельный модуль `core/prompt.py` с `PROMPT_VERSION` (строка, семвер-подобная `v1`); логирование версии в meta — при записи messages в 3.7.
   - **Allowlist для eval и прода (результат приёмки 2026-07-10)**: официального провайдера DeepSeek для deepseek-chat на OpenRouter нет; доступны StreamLake (квант неизвестен), DeepInfra (fp4), Novita (fp8). Принято: `OPENROUTER_ALLOWED_PROVIDERS=novita` (fp8 — старший квант) для eval и как прод-дефолт до пересмотра в 3.5/3.6. Разброс между хостерами подтверждён эмпирически: без пина два прогона дали 18/20 и 13/20 с несовпадающими провалами; на пине novita — стабильные 20/20 дважды.
   - **Eval**: `evals/prompt_eval.py` + сценарии в `evals/scenarios_v1.py` (20 сценариев по разделу 4 спецификации) — живой прогон на deepseek-chat через OpenRouterProvider, в CI и pytest НЕ входит (тратит токены; запуск ручной при каждой правке промпта). Инструменты в eval — только ToolDefinition-объявления из реестра (send_email, remember_fact, fetch_page и т.д.), без реализаций: проверяется выбор и формулировки, не исполнение. Оценка сценария — программные ассерты (вызван ли нужный инструмент / отсутствие вызова / подстроки в ответе), не LLM-судья.
3.5. Глобальный семафор конкурентных запросов к OpenRouter (Redis) + retry с экспоненциальной задержкой на 429/5xx. Конфиг fallback-модели той же семьи (напр. deepseek-chat ↔ V4 Flash): переключение после N подряд неудач по всем провайдерам основной, с алертом админу. Детализация (2026-07-10):
   - **Форма**: обёртка `ResilientLLMProvider` (реализует LLMProvider, композиция вокруг внутреннего провайдера) в `core/llm_resilient.py` — агентный цикл и воркер её не различают. Семафор/брейкер живут в redis-queue (не cache: allkeys-lru может вытеснить счётчики).
   - **Семафор**: глобальный лимит конкурентных generate() ко всем воркерам, ключ `sem:openrouter` — Lua INCR с потолком и TTL 180 с на «утёкшие» слоты (декремент в finally; TTL — страховка от смерти воркера). Лимит из env `OPENROUTER_MAX_CONCURRENCY`, дефолт 8. Занято → ждать с шагом 0.2 с до 30 с, затем LLMProviderError (задача переиграется очередью).
   - **Retry**: на `LLMRateLimitError`/`LLMServerError` (уже есть в core/llm.py) — до 3 повторов, base 1 с, множитель 2, полный джиттер; слот семафора на время сна не удерживать.
   - **Fallback + circuit breaker**: env `OPENROUTER_FALLBACK_MODEL` (дефолт пуст = выключено). Счётчик подряд неудач основной модели (после всех ретраев) в Redis `cb:openrouter:{model}`; ≥3 → все запросы идут на fallback в течение cooldown 300 с, после — проба основной (успех сбрасывает счётчик). Первое срабатывание брейкера — алерт админу через существующий notify_admin-канал воркера (колбэк прокидывается в обёртку опционально; повторные алерты не чаще раза в cooldown).
   - Значения лимитов/таймаутов — конфиг с дефолтами выше; тесты — MockLLMProvider с инжектируемыми ошибками и fake-clock/sleep, редис-часть — интеграционные тесты на локальном redis.
3.6. Роутинг моделей: v1 — одна модель (deepseek-chat); интерфейс роутера заложить (вход: тип задачи, план пользователя → модель + множитель бюджета). [Обсудить: правила роутинга при добавлении выбора моделей] Детализация (2026-07-10, вопрос №7 отправлен владельцу — до его решения правила не пишем, только интерфейс):
   - `core/model_router.py`: `RouteRequest` (task_type: Literal["interactive","background"], user_plan: str) → `ModelRoute` (model: str, budget_multiplier: Decimal); протокол `ModelRouter` + v1-реализация `StaticModelRouter(model)` — всегда основная модель из настроек OpenRouter, множитель 1.
   - Вшивка: воркер строит маршрут перед созданием AgentLoop; `AgentLoopConfig.task_budget_rub` умножается на множитель маршрута (v1 — без эффекта); выбранная моделью маршрута строка на v1 обязана совпадать с settings.model (assert-проверка конфигурации при старте).
   - Тесты юнит; правила роутинга при второй модели — после решения №7.
3.7. Персистентность диалога: запись сообщений и ответов в messages в конце задачи; суммаризации — в dialog_summaries (схема — [db-schema.md](db-schema.md)).
3.8. Учёт стоимости: токены, cached_tokens и $ per-запрос — в таблицу usage (схема — [db-schema.md](db-schema.md)).

[DoD] Диалог с памятью работает end-to-end с одним тестовым инструментом; стоимость каждой задачи видна в БД; обрыв воркера не теряет задачу (переигрывается из stream).
