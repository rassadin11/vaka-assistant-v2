# Codex Review — независимое ревью

## Когда НЕ использовать

- Мелкие локальные проверки — обычное ревью силами Claude быстрее и не хуже.
- Не делегировать ревью только для того, чтобы самому не читать код: Claude обязан прочитать дифф в любом случае.
- Вывод Codex — свидетельство, а не приговор: каждое замечание оценить и явно согласиться или возразить.

## 0. Проверка доступности

`codex --version`. Если CLI нет — сообщить пользователю: `npm install -g @openai/codex`, затем `codex login`.

## 1. Составление промпта

Codex не видит наш диалог. Промпт самодостаточный:

- что ревьюим: конкретные файлы/дифф (абсолютные пути) и против чего проверяем — ссылки на спеку (`plan/stage-N.md`, DoD, `plan/db-schema.md`);
- критерии проекта, релевантные задаче: инварианты из `CLAUDE.md` (RLS с `SET LOCAL`, идемпотентность mutating-инструментов, запрет `user_id` из аргументов LLM, мок-провайдеры в тестах);
- формат ответа: вердикт + находки, отсортированные по серьёзности, каждая с файлом/строкой и объяснением, чем это грозит.

Секреты в промпт не включать.

## 2. Запуск (только чтение)

```
codex exec --sandbox read-only --skip-git-repo-check --cd "D:\claude-projects\personal-assistant" "<промпт>"
```

- Запуск может превышать дефолтный таймаут Bash (2 мин) — передавать явный `timeout` (до 600000 мс) или запускать в фоне.
- Длинный промпт — через файл в scratchpad: `codex exec ... "$(cat <файл>)"`.
- Объёмное ревью, вывод которого не нужен в основном контексте целиком, — через сабагента `codex-runner`.

## 3. После выполнения

1. Пройтись по каждой находке: подтвердить по коду или мотивированно отклонить. Не пересказывать список без собственной оценки.
2. Подтверждённые проблемы — в работу по обычному регламенту (сначала план, потом код, если затронута архитектура).
3. Итог пользователю: что ревьюил Codex, его вердикт, моя оценка (согласен/не согласен и почему), что делаем дальше.
